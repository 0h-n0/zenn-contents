### 実装例：KPIダッシュボード

以下は、Node.js + GitHub Copilot APIを使ったKPI測定の最小実装例です：

```typescript
// kpi-tracker.ts
import { Copilot } from '@github/copilot-api';

interface KPI {
  taskSuccessRate: number;
  avgResponseTime: number;
  codeAcceptanceRate: number;
  costSavings: number;
}

async function measureCopilotKPI(): Promise<KPI> {
  const copilot = new Copilot({ apiKey: process.env.GITHUB_TOKEN });

  // 過去24時間のメトリクス取得
  const metrics = await copilot.getUsageMetrics({
    period: '24h'
  });

  return {
    taskSuccessRate: metrics.completions / metrics.requests, // 成功率
    avgResponseTime: metrics.avgLatency, // 平均レスポンス時間（ms）
    codeAcceptanceRate: metrics.acceptedSuggestions / metrics.totalSuggestions, // 提案受諾率
    costSavings: calculateSavings(metrics.timeSaved) // 節約時間から換算
  };
}

function calculateSavings(timeSaved: number): number {
  const hourlyRate = 5000; // エンジニア1時間あたりのコスト（円）
  return (timeSaved / 3600) * hourlyRate;
}

// 使用例
const kpi = await measureCopilotKPI();
console.log(`成功率: ${kpi.taskSuccessRate * 100}%`);
console.log(`平均応答時間: ${kpi.avgResponseTime}ms`);
console.log(`コスト削減: ¥${kpi.costSavings}/日`);
```

**なぜこの実装を選んだか:**
- 公式APIを使用することで、正確なメトリクスを取得
- 時間単位の集計により、日次・週次レポートに柔軟に対応
- コスト換算により、経営層への報告資料に活用可能

**注意点:**
> このコードは GitHub Copilot Business/Enterprise プランでのみ動作します。個人プランではメトリクスAPIが利用できません。

## よくある失敗と対処法

### 失敗パターン1: AIに丸投げして品質低下

**状況**: GitHub Copilotの提案をレビューなしで受け入れ、セキュリティホールが本番環境へ

**原因**: 「AIが生成したコードは正しい」という過信

**対処法**:
- **必ずコードレビューを実施**（人間の目でチェック）
- **静的解析ツール導入**（SonarQube、Semgrep）
- **セキュリティスキャン自動化**（Dependabot、Snyk）

### 失敗パターン2: KPI設計の誤り

**状況**: 「コード行数が増えた = 生産性向上」と誤解

**原因**: 量的指標に偏り、品質指標を無視

**対処法**:
- **リードタイム測定**: 機能要望から本番リリースまでの時間
- **変更失敗率**: デプロイ後のロールバック頻度
- **顧客満足度**: NPS（Net Promoter Score）

### 失敗パターン3: 組織文化の抵抗

**状況**: 「AIに仕事を奪われる」という不安から現場が抵抗

**原因**: トップダウンでの強制導入

**対処法**:
- **段階的導入**: パイロットチームから開始
- **成功事例の共有**: 社内勉強会で効果を可視化
- **役割の再定義**: 「AIが書くコードをレビューする」新しい役割
