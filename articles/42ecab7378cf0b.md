## クエリ最適化パイプラインを実装する

ユーザーが入力するクエリは、しばしば曖昧で短く、検索に必要な情報が不足しています。クエリ最適化は、この「検索意図と文書表現のギャップ」を埋める前処理です。ここでは3つのテクニックとその組み合わせ方を実装します。

### Multi-Query Rewriteでクエリを多面的に展開する

Multi-Query Rewrite は、1つのユーザークエリを複数の視点に書き換え、それぞれで検索して結果を統合する手法です。ユーザーの意図を1つの表現に限定しないことで、**検索の網羅性（Recall）を改善**できます。

```python
# multi_query_rewrite.py
from openai import OpenAI

client = OpenAI()


def generate_multi_queries(original_query: str, n_queries: int = 3) -> list[str]:
    """元のクエリから複数の検索クエリを生成する。

    Args:
        original_query: ユーザーの元のクエリ
        n_queries: 生成するクエリ数

    Returns:
        生成されたクエリのリスト（元のクエリを含む）
    """
    prompt = f"""以下のユーザークエリに対して、異なる視点から{n_queries}個の検索クエリを生成してください。
各クエリは改行で区切ってください。元のクエリの意図を保ちつつ、異なる語彙や観点を使ってください。

ユーザークエリ: {original_query}

検索クエリ:"""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.7,
        max_tokens=256,
    )
    generated = response.choices[0].message.content.strip().split("\n")
    queries = [q.strip().lstrip("0123456789.-) ") for q in generated if q.strip()]
    return [original_query] + queries[:n_queries]
```

**なぜこの実装を選んだか:**
- `gpt-4o-mini`を使うことでコストを抑えつつ十分な品質のクエリ生成が可能
- 元のクエリを必ずリストに含めることで、リライトが失敗しても最低限の検索品質を保証
- `temperature=0.7`は多様性と品質のバランスとして、複数のRAGフレームワークで採用されている値

**注意点:**
> Multi-Queryは検索回数が増えるためレイテンシとAPIコストが比例して増加します。クエリ数は2〜4程度に抑え、結果の統合にはRRF（Reciprocal Rank Fusion）を使うことを推奨します。

### クエリ分解で複合質問に対応する

「Pythonで非同期処理を使ったWebスクレイピングのエラーハンドリング方法」のような複合質問は、1つのベクトルでは検索意図を十分に捉えられません。クエリ分解は、複合質問をサブクエリに分割して個別に検索する手法です。

```python
# query_decomposition.py
from openai import OpenAI

client = OpenAI()


def decompose_query(complex_query: str) -> list[str]:
    """複合質問をサブクエリに分解する。

    Args:
        complex_query: 複合的なユーザークエリ

    Returns:
        分解されたサブクエリのリスト
    """
    prompt = f"""以下の質問を、それぞれ独立して検索可能なサブ質問に分解してください。
各サブ質問は1行ずつ出力してください。分解が不要な単純な質問の場合はそのまま返してください。

質問: {complex_query}

サブ質問:"""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.0,
        max_tokens=256,
    )
    sub_queries = response.choices[0].message.content.strip().split("\n")
    return [q.strip().lstrip("0123456789.-) ") for q in sub_queries if q.strip()]
```

### HyDEの条件付きフォールバックで安定運用する

HyDE（Hypothetical Document Embeddings）は、クエリからLLMで仮想的な回答文書を生成し、その文書のEmbeddingで検索する手法です。短いクエリに対して有効で、HyPE（HyDEの発展形）では**Precisionが最大42ポイント改善**したと報告されています。

しかし、HyDEには重大な落とし穴があります。**LLMが誤った仮想文書を生成した場合、検索精度が逆に悪化します**。特に専門ドメインやLLMの知識が不足する領域で顕著です。

実運用では、HyDEを常時適用するのではなく、**通常の検索の信頼度が低いときだけHyDEにフォールバックする条件付き戦略**が有効です。

```python
# hyde_conditional.py
from dataclasses import dataclass

import numpy as np
from openai import OpenAI
from sentence_transformers import SentenceTransformer

client = OpenAI()
encoder = SentenceTransformer("intfloat/multilingual-e5-large-instruct")


@dataclass
class SearchResult:
    text: str
    score: float


def compute_confidence(scores: list[float], threshold: float = 0.75) -> bool:
    """検索結果の信頼度を判定する。

    上位結果のスコアが閾値を超え、かつ上位と下位の差が
    十分にある場合を「高信頼」と判定する。

    Args:
        scores: 検索スコアのリスト（降順）
        threshold: 信頼度の閾値

    Returns:
        高信頼ならTrue
    """
    if not scores:
        return False
    top_score = scores[0]
    score_gap = scores[0] - scores[-1] if len(scores) > 1 else 0
    return top_score >= threshold and score_gap >= 0.1


def generate_hypothetical_document(query: str) -> str:
    """HyDE: クエリから仮想文書を生成する。"""
    prompt = f"""以下の質問に対する回答を含む技術文書の一節を生成してください。
200文字程度で、具体的な技術情報を含めてください。

質問: {query}

文書:"""
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.3,
        max_tokens=300,
    )
    return response.choices[0].message.content.strip()


def search_with_conditional_hyde(
    query: str,
    documents: list[str],
    doc_embeddings: np.ndarray,
    top_k: int = 10,
    confidence_threshold: float = 0.75,
) -> list[SearchResult]:
    """条件付きHyDEで検索する。

    通常の検索を先に行い、信頼度が低い場合のみHyDEを適用する。

    Args:
        query: 検索クエリ
        documents: 文書リスト
        doc_embeddings: 文書Embeddingの行列
        top_k: 返す結果数
        confidence_threshold: HyDEフォールバックの閾値
    """
    # Step 1: 通常の検索
    query_embedding = encoder.encode(f"query: {query}", normalize_embeddings=True)
    scores = np.dot(doc_embeddings, query_embedding)
    top_indices = np.argsort(scores)[::-1][:top_k]
    top_scores = [float(scores[i]) for i in top_indices]

    # Step 2: 信頼度チェック → 低ければHyDEにフォールバック
    if not compute_confidence(top_scores, confidence_threshold):
        hypothetical_doc = generate_hypothetical_document(query)
        hyde_embedding = encoder.encode(
            f"passage: {hypothetical_doc}", normalize_embeddings=True
        )
        # 元クエリとHyDE Embeddingの加重平均
        blended = 0.6 * query_embedding + 0.4 * hyde_embedding
        blended = blended / np.linalg.norm(blended)
        scores = np.dot(doc_embeddings, blended)
        top_indices = np.argsort(scores)[::-1][:top_k]

    return [
        SearchResult(text=documents[i], score=float(scores[i]))
        for i in top_indices
    ]
```

**なぜ条件付きにするのか:**
- HyDEのLLM呼び出しは100〜300msのレイテンシ増加を伴う
- 通常検索で十分な精度が出ているケースでHyDEを適用すると、逆にノイズが混入するリスクがある
- 加重平均（0.6:0.4）により、HyDEの仮想文書が不正確でも元クエリの情報を保持できる

**ハマりポイント:**
> `confidence_threshold`の値はデータセットに大きく依存します。初期値0.75で開始し、後述の評価ループで最適値を探索してください。公式ドキュメントによると、HyDEは「答えが存在しない質問」に対して仮の答えを生成してしまい、検索精度が悪化するケースがあります。FAQ検索や明確な回答が存在するドメインに向いています。

## 多段リランキングアーキテクチャを構築する

初期検索（bi-encoder）で取得した候補を、より精密なモデルで並べ替えるリランキングは、セマンティック検索精度向上の中核テクニックです。ここでは**2段階のリランキングパイプライン**を実装します。

### 2段階リランキングの全体設計

多段リランキングの基本思想は、**コストの低い手法で候補を絞り、コストの高い手法で精密に並べ替える**ことです。

```mermaid
graph LR
    A[ユーザークエリ] --> B[bi-encoder検索<br/>Top 100取得]
    B --> C[cross-encoder<br/>Top 100→Top 20]
    C --> D[最終結果<br/>Top 10]
```

| ステージ | モデル例 | 処理対象 | レイテンシ |
|---------|---------|---------|-----------|
| 初期検索 | intfloat/multilingual-e5-large | 全文書 → Top 100 | 10〜50ms |
| Stage 1 | BAAI/bge-reranker-v2-m3 | Top 100 → Top 20 | 50〜100ms |
| Stage 2（任意） | LLMジャッジ | Top 20 → Top 10 | 200〜500ms |

### cross-encoderリランカーの実装

cross-encoderは、クエリと文書のペアを同時に処理して関連度スコアを算出します。bi-encoderが個別にエンコードしてコサイン類似度で比較するのに対し、cross-encoderはペア全体の文脈を考慮できるため、**より精密な関連度判定**が可能です。

```python
# reranker_pipeline.py
from dataclasses import dataclass

import torch
from transformers import AutoModelForSequenceClassification, AutoTokenizer


@dataclass
class RankedResult:
    text: str
    initial_score: float
    rerank_score: float


class CrossEncoderReranker:
    """cross-encoderベースのリランカー。

    BAAI/bge-reranker-v2-m3を使用。100以上の言語をサポートし、
    Apache 2.0ライセンスで商用利用可能。

    動作確認: Python 3.11, transformers 4.47, torch 2.5
    """

    def __init__(
        self,
        model_name: str = "BAAI/bge-reranker-v2-m3",
        device: str | None = None,
    ):
        self.device = device or ("cuda" if torch.cuda.is_available() else "cpu")
        self.tokenizer = AutoTokenizer.from_pretrained(model_name)
        self.model = AutoModelForSequenceClassification.from_pretrained(model_name)
        self.model.to(self.device)
        self.model.eval()

    def rerank(
        self,
        query: str,
        documents: list[SearchResult],
        top_k: int = 20,
    ) -> list[RankedResult]:
        """検索結果をcross-encoderでリランキングする。

        Args:
            query: 検索クエリ
            documents: 初期検索結果のリスト
            top_k: 返す結果数

        Returns:
            リランキングされた結果のリスト
        """
        pairs = [[query, doc.text] for doc in documents]

        with torch.no_grad():
            inputs = self.tokenizer(
                pairs,
                padding=True,
                truncation=True,
                max_length=512,
                return_tensors="pt",
            ).to(self.device)
            scores = self.model(**inputs).logits.squeeze(-1).cpu().tolist()

        ranked = [
            RankedResult(
                text=doc.text,
                initial_score=doc.score,
                rerank_score=score,
            )
            for doc, score in zip(documents, scores)
        ]
        ranked.sort(key=lambda x: x.rerank_score, reverse=True)
        return ranked[:top_k]
```

**なぜBGE-reranker-v2-m3を選んだか:**
- Apache 2.0ライセンスで商用利用に制約がない
- 100以上の言語をサポートしており、日本語ドキュメントにも対応
- 278Mパラメータで、GPU利用時50〜100msのレイテンシに収まる
- Agentsetのベンチマークでは、商用APIモデルと比較して遜色ない精度を達成

**トレードオフ:**
- cross-encoderはペアごとに推論するため、候補数に比例してレイテンシが増加します。Top 100を処理する場合、GPU利用でも100〜200msかかります
- `max_length=512`でトークンを切り詰めているため、長い文書では末尾の情報が失われます。長文対応が必要な場合はチャンク分割との併用を検討してください

### 2026年時点のリランカーモデル比較

本番環境でのモデル選定の参考として、2026年3月時点の主要リランカーを比較します。

|-------|------|-----------|-----------|-----------|-------|
| zerank-1 | 商用API | +28% (対ベースライン) | 中速 | 商用 | 対応 |
| Voyage Rerank 2.5 | 商用API | zerank-1と同等 | zerank-1の約1/2 | 商用 | 対応 |

Agentsetのベンチマークによると、**zerank-1が総合ELOスコアで最高**を記録し、**Voyage Rerank 2.5がレイテンシと精度のバランスで最も実用的**と評価されています。一方、**特定ドメイン（科学・法律）ではドメイン特化モデルが勝つケース**もあり、自社データでの評価が不可欠です。

**選定の判断基準:**
- **コスト重視・自社GPU利用可能** → BGE-reranker-v2-m3
- **レイテンシ重視の本番API** → Voyage Rerank 2.5
- **精度最優先** → zerank-1
- **すでにCohereを利用中** → Cohere Rerank 3.5（エコシステム統合のメリット）

## 評価駆動チューニングループを回す

検索精度の改善は「施策を実装→計測→比較→次の施策を決定」のループで進めます。主観的な「良くなった気がする」ではなく、**定量指標に基づく改善サイクル**がカギです。

### 検索評価指標を理解する

セマンティック検索で使われる主要な評価指標を整理します。

| 指標 | 計算方法 | 用途 |
|------|---------|------|
| **NDCG@k** | 上位k件の関連度を理想的な順位と比較 | 順位の質を総合評価 |
| **MRR** | 最初の正解文書の順位の逆数 | 「1件目の当たり」の速さ |
| **Recall@k** | 上位k件に含まれる正解の割合 | 取りこぼしの少なさ |
| **Precision@k** | 上位k件のうち正解の割合 | ノイズの少なさ |

RAGパイプラインでは、LLMに渡すコンテキストの質が重要なため、**NDCG@10とRecall@10の組み合わせ**で評価するのが一般的です。

### 評価パイプラインの実装

評価を自動化するパイプラインを実装します。正解データ（クエリとそれに対する正解文書のペア）が必要です。

```python
# search_evaluator.py
import json
from dataclasses import dataclass
from pathlib import Path

import numpy as np


@dataclass
class EvalQuery:
    query: str
    relevant_doc_ids: list[str]  # 正解文書のIDリスト


@dataclass
class EvalResult:
    ndcg_at_10: float
    mrr: float
    recall_at_10: float
    precision_at_10: float


def dcg_at_k(relevances: list[int], k: int) -> float:
    """DCG@kを計算する。"""
    relevances = relevances[:k]
    return sum(rel / np.log2(i + 2) for i, rel in enumerate(relevances))


def ndcg_at_k(relevances: list[int], k: int) -> float:
    """NDCG@kを計算する。

    Args:
        relevances: 各順位の関連度（1=関連, 0=非関連）
        k: 評価する上位件数

    Returns:
        NDCG@kスコア（0.0〜1.0）
    """
    actual_dcg = dcg_at_k(relevances, k)
    ideal_relevances = sorted(relevances, reverse=True)
    ideal_dcg = dcg_at_k(ideal_relevances, k)
    if ideal_dcg == 0:
        return 0.0
    return actual_dcg / ideal_dcg


def evaluate_search(
    eval_queries: list[EvalQuery],
    search_fn: callable,
    k: int = 10,
) -> EvalResult:
    """検索パイプライン全体を評価する。

    Args:
        eval_queries: 評価用クエリと正解のリスト
        search_fn: 検索関数（query -> list[doc_id]）
        k: 評価する上位件数

    Returns:
        各指標の平均スコア
    """
    ndcg_scores = []
    mrr_scores = []
    recall_scores = []
    precision_scores = []

    for eq in eval_queries:
        retrieved_ids = search_fn(eq.query)[:k]
        relevant_set = set(eq.relevant_doc_ids)

        # 関連度リスト作成
        relevances = [1 if doc_id in relevant_set else 0 for doc_id in retrieved_ids]

        # NDCG@k
        ndcg_scores.append(ndcg_at_k(relevances, k))

        # MRR
        mrr = 0.0
        for i, doc_id in enumerate(retrieved_ids):
            if doc_id in relevant_set:
                mrr = 1.0 / (i + 1)
                break
        mrr_scores.append(mrr)

        # Recall@k
        retrieved_relevant = len(set(retrieved_ids) & relevant_set)
        recall = retrieved_relevant / len(relevant_set) if relevant_set else 0.0
        recall_scores.append(recall)

        # Precision@k
        precision = retrieved_relevant / k if k > 0 else 0.0
        precision_scores.append(precision)

    return EvalResult(
        ndcg_at_10=float(np.mean(ndcg_scores)),
        mrr=float(np.mean(mrr_scores)),
        recall_at_10=float(np.mean(recall_scores)),
        precision_at_10=float(np.mean(precision_scores)),
    )


def load_eval_dataset(path: str) -> list[EvalQuery]:
    """評価データセットをJSONから読み込む。

    期待するフォーマット:
    [{"query": "...", "relevant_doc_ids": ["id1", "id2"]}, ...]
    """
    data = json.loads(Path(path).read_text())
    return [EvalQuery(query=d["query"], relevant_doc_ids=d["relevant_doc_ids"]) for d in data]
```

### 評価データセットの作り方

評価ループの最大の障壁は「正解データをどう用意するか」です。以下の3つのアプローチがあります。

**1. LLM-as-Judgeで半自動生成する**

既存のドキュメントからLLMでクエリと正解ペアを生成する方法です。コストが低く、大量の評価データを素早く作れますが、LLMの知識に偏りがある点に注意が必要です。

```python
# generate_eval_data.py
from openai import OpenAI

client = OpenAI()


def generate_eval_pairs(document: str, doc_id: str, n_queries: int = 3) -> list[dict]:
    """文書からクエリ-正解ペアを生成する。

    Args:
        document: 元の文書テキスト
        doc_id: 文書ID
        n_queries: 生成するクエリ数

    Returns:
        {"query": str, "relevant_doc_ids": [str]} のリスト
    """
    prompt = f"""以下の文書を読み、この文書を検索で見つけるために使いそうな自然な質問を{n_queries}個作成してください。
各質問は改行で区切ってください。質問は多様な表現を使い、文書の異なる側面をカバーしてください。

文書:
{document[:2000]}

質問:"""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.7,
    )
    queries = response.choices[0].message.content.strip().split("\n")
    return [
        {"query": q.strip().lstrip("0123456789.-) "), "relevant_doc_ids": [doc_id]}
        for q in queries
        if q.strip()
    ]
```

**2. 検索ログから抽出する**

本番環境のユーザー検索ログとクリックデータから、暗黙的な関連度フィードバックを構築します。実際のユーザー行動に基づくため信頼性が高いですが、クリックバイアス（上位に表示された結果がクリックされやすい）に注意が必要です。

**3. ドメインエキスパートがアノテーションする**

最も信頼性が高いですが、コストも高い方法です。50〜100件のクエリに対して正解を付与するだけでも、方向性の検証には十分です。

**制約条件:**
> LLM-as-Judgeで生成した評価データは、実際のユーザークエリとは分布が異なる可能性があります。本番投入前に、必ずドメインエキスパートによるサンプルチェック（全体の10〜20%）を実施してください。

### パラメータチューニングの実践

評価パイプラインが整ったら、以下のパラメータを系統的にチューニングします。

| パラメータ | 調整範囲 | 影響 |
|-----------|---------|------|
| 初期検索のTop-k | 50〜200 | Recallとリランキングコストのトレード |
| リランカーのTop-k | 10〜30 | Precisionとコンテキスト量のトレード |
| HyDEフォールバック閾値 | 0.6〜0.85 | HyDE適用頻度とレイテンシ |
| Multi-Queryのクエリ数 | 2〜4 | Recallとレイテンシ |
| HyDE加重平均の比率 | 0.3〜0.7 | 元クエリ vs 仮想文書の影響度 |

**よくある間違い:**
最初にすべてのパラメータを同時に変更してしまうケースがあります。**1パラメータずつ変更して効果を計測**し、改善が確認できたら固定して次に進む「グリーディ探索」が実用的です。

## よくある問題と解決方法

| 問題 | 原因 | 解決方法 |
|------|------|----------|
| リランキング後にRecallが低下 | 初期検索のTop-kが小さすぎる | Top-kを100以上に拡大 |
| HyDEで検索精度が悪化 | 仮想文書がドメイン外 | 条件付きフォールバック + プロンプト改善 |
| Multi-Queryで重複結果が多い | クエリの多様性が不足 | temperatureを上げるか、明示的な多様性指示を追加 |
| 評価スコアが安定しない | 評価データが少なすぎる | 最低50クエリ以上で評価 |
| cross-encoderのレイテンシが高い | CPU利用 or 候補数過多 | GPU利用 + 初期検索のTop-kを削減 |
| NDCG@10は高いがユーザー満足度が低い | 指標と実際の検索体験の乖離 | MRR（1件目の精度）も併用、ユーザーフィードバック収集 |

## まとめと次のステップ

**まとめ:**
- **クエリ最適化**（Multi-Query Rewrite、クエリ分解、条件付きHyDE）で検索意図と文書表現のギャップを埋める
- **多段リランキング**（bi-encoder → cross-encoder）で初期検索結果の順位精度を改善する。2026年3月時点では、OSSならBGE-reranker-v2-m3、商用APIならzerank-1またはVoyage Rerank 2.5が有力
- **評価駆動チューニング**（NDCG@10、MRR、Recall@k）で改善効果を定量化し、パラメータを系統的に最適化する
- すべてのテクニックを一度に導入せず、**cross-encoderリランキング → 評価ループ構築 → クエリ最適化追加**の順で段階的に進めるのがコスト対効果で合理的

**次にやるべきこと:**
- 自社データで50件以上の評価データセットを作成し、現在のベースラインNDCG@10を計測する
- BGE-reranker-v2-m3またはVoyage Rerank 2.5でリランキングを追加し、改善幅を計測する
- 改善幅が不十分な場合にMulti-Query RewriteやHyDE条件付きフォールバックを追加する
