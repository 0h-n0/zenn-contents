## LLM評価のパラダイムシフト：統計指標からLLM-as-a-Judgeへ

### 従来の統計指標の限界

BLEU、ROUGE、METEORといった従来の統計指標は、n-gramの重複や編集距離を測定しますが、**意味的理解や推論能力がほぼゼロ**です。これらの指標は、スペルチェックのような特化タスクでのみ有効で、複雑なLLM出力の評価には不適切です。

```python
from nltk.translate.bleu_score import sentence_bleu

# 従来のBLEU評価（セマンティクスを無視）
reference = [["the", "cat", "is", "on", "the", "mat"]]
candidate = ["the", "cat", "sat", "on", "the", "mat"]
score = sentence_bleu(reference, candidate)  # 0.75（"sat"の違いしか検出できない）
```

**なぜ従来手法では不十分か:**
- 意味が同じでも表現が異なると低スコア（"happy" vs "joyful"）
- 文脈を無視した評価（前後の文章との関連性を考慮しない）
- 創造的な出力を正しく評価できない（小説生成、詩作等）

### LLM-as-a-Judge：評価の民主化

LLM-as-a-Judgeは、**LLM自身を評価者として使用**する革新的アプローチです。人間と同等の自然言語理解能力を持つLLMに評価基準を明示的に指示することで、複雑なタスクの自動評価が可能になります。

```python
from openai import OpenAI

client = OpenAI()

def evaluate_response(user_query: str, llm_output: str) -> dict:
    """LLM-as-a-Judge実装例"""
    evaluation_prompt = f"""
    以下の観点でLLMの回答を1-5で評価してください:

    【評価基準】
    1. 関連性: ユーザー質問に適切に答えているか
    2. 正確性: 事実に基づいた情報か
    3. 有用性: 実用的で具体的な内容か
    4. 簡潔性: 冗長な説明を避けているか

    ユーザー質問: {user_query}
    LLM回答: {llm_output}

    JSON形式で出力:
    {{"relevance": 点数, "accuracy": 点数, "usefulness": 点数, "conciseness": 点数, "reason": "理由"}}
    """

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": evaluation_prompt}],
        response_format={"type": "json_object"}
    )

    return response.choices[0].message.content
```

## 本番運用で直面する4つのバイアスと対策

LLM-as-a-Judgeは強力ですが、**体系的なバイアス**を持ちます。本番環境で95%以上の信頼性を確保するには、これらの対策が必須です。

### 1. Position Bias（順序バイアス）

評価対象の提示順序で結果が変わる問題。A→B と B→A で評価が逆転するケースが30-40%存在します。

**対策: ランダムシャッフル + 複数回評価**

```python
import random
from statistics import mean

def robust_comparison(response_a: str, response_b: str, trials: int = 3) -> str:
    """Position Biasを軽減する複数回評価"""
    results = []

    for _ in range(trials):
        # ランダムに順序を入れ替え
        if random.random() > 0.5:
            score = evaluate_pair(response_a, response_b)
            results.append("A" if score > 0 else "B")
        else:
            score = evaluate_pair(response_b, response_a)
            results.append("B" if score > 0 else "A")

    # 多数決で最終判定
    return max(set(results), key=results.count)
```

### 2. Length Bias（長さバイアス）

長い回答を過大評価する傾向。詳細な説明が必要ない質問でも、長文を高スコアにしてしまいます。

**対策: 明示的な長さ制約 + 簡潔性スコア**

```python
evaluation_rubric = """
【簡潔性の評価基準】
5点: 必要最小限の情報で完結（100-200文字）
3点: やや冗長だが許容範囲（200-400文字）
1点: 過度に冗長（400文字以上で削減可能）

※ユーザー質問が「簡単な説明」を求めている場合、長文は減点対象
"""
```

### 3. Concreteness Bias（具体性バイアス）

数値・引用・権威を含む回答を過大評価。定性的な説明が適切な場合でも、無意味な数値を追加すると高評価になります。

**対策: タスク特性に応じた評価基準設計**

```python
task_specific_rubric = {
    "factual_qa": "統計や出典を含む回答を高評価",
    "creative_writing": "具体的数値は不要。創造性と表現力を重視",
    "code_generation": "実行可能性とエッジケース対応を評価"
}
```

### 4. Self-Enhancement Bias（自己強化バイアス）

同じモデルファミリーの出力を高評価する傾向。GPT-4で評価すると、GPT-4生成の回答が優遇されます。

**対策: 異なるモデルで評価 + クロスバリデーション**
