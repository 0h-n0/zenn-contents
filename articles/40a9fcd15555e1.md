## Agent Harnessとは何か

**Agent Harness（エージェント・ハーネス）**とは、AIモデルのライフサイクル・コンテキスト・外部世界との対話を管理するソフトウェアインフラです。AIエージェント自体が「推論・意思決定を行うLLM」であるのに対し、Agent Harnessは「そのエージェントを制御・監視・最適化する外側のシステム」として機能します。

| 要素 | AIエージェント | Agent Harness |
|------|----------------|---------------|
| 役割 | 推論・意思決定 | 管理・制御・監視 |
| 技術要素 | LLM、プロンプト | コンテキスト管理、セッション制御 |
| 責務 | タスク実行 | 信頼性・安定性・永続性 |
| 比喩 | 車のエンジン | 車体・ハンドル・ブレーキ |

**なぜ今Agent Harnessが重要か:**
- **Model Drift（モデル漂流）の解決**: コンテキストが100ステップ超えるとLLMの指示追従能力が低下する問題を、Harnessで検出・修正
- **長時間実行タスクの実現**: 複数セッションにまたがる開発タスク（数時間〜数日）を可能にする
- **本番運用の信頼性**: 企業システムで求められるセキュリティ・監査・エラーハンドリングを提供

## AnthropicのLong-Running Agent Harness

長時間実行エージェントの最大の課題は、**各新規セッションが前回のセッションを記憶していない**ことです。Anthropicは、この問題を**初期化エージェント + コーディングエージェント**の2層構造で解決しました。

```python
# Anthropic Agent Harness 概念的な実装例（疑似コード）
from anthropic import Anthropic

client = Anthropic(api_key="your-api-key")

# 1. 初期化エージェント（初回実行のみ）
def initialize_environment(project_goal: str):
    initializer = client.agents.create(
        model="claude-sonnet-4-5",
        system_prompt="features.md, .git, progress.json, context.mdを作成",
        tools=["write_file", "run_shell", "git_commit"]
    )
    return AgentSession(agent=initializer).run(project_goal).artifacts

# 2. コーディングエージェント（繰り返し実行）
def coding_session(task_description: str, previous_artifacts: dict):
    coder = client.agents.create(
        model="claude-sonnet-4-5",
        system_prompt="features.mdから1タスク選択、実装、テスト、progress.json更新",
        tools=["read_file", "edit_file", "run_tests", "git_commit"],
        context=previous_artifacts
    )
    return AgentSession(agent=coder, max_steps=50).run(task_description)

# 実行例: 10セッションで段階的に実装
artifacts = initialize_environment("Build a web app for task management")
for i in range(10):
    result = coding_session(f"Implement feature #{i+1}", artifacts)
    artifacts.update(result.artifacts)
```

**2層構造の利点:**
- **明確な責務分離**: 初期化（1回）と作業（繰り返し）を分離
- **進捗の可視化**: progress.jsonで人間が進捗を把握可能
- **max_steps=50でModel Drift防止**

**トラブルシューティング:**

| 問題 | 原因 | 解決方法 |
|------|------|----------|
| コンテキストオーバー | 1セッションで多くをやりすぎ | max_steps=50で強制区切り |
| 進捗不明 | progress.jsonが更新されない | 各セッション終了時にファイル更新を必須化 |
| Git履歴が汚い | エージェントが過度にコミット | コミットフックで1セッション=1コミットに制限 |

## Context Engineeringの3原則

Agent Harnessの設計で最も重要な概念が**Context Engineering（コンテキスト工学）**です。

### 原則1: Compression（圧縮）

LLMに「すべての情報」を渡すのではなく、**必要な情報だけを要約して渡す**戦略です。5万文字のログを300文字に圧縮することで、コンテキストウィンドウを節約し、トークン使用量を1/100以下に削減できます。

### 原則2: Injection（RAG注入）

必要な情報を**必要な時だけ**コンテキストに注入する戦略です。RAG（検索拡張生成）でベクトル検索を使い、1,000ページのドキュメントから関連するTop 3のみ注入することで、精度向上とコスト削減を両立します。

### 原則3: Isolation（サブエージェント分離）

複雑なタスクを**複数の専門サブエージェント**に分離する戦略です。

```python
# 概念的な実装例（疑似コード）
from anthropic import Anthropic

client = Anthropic(api_key="your-api-key")

# 設計・実装・レビューで異なるモデルを使用（疑似コード）
design_agent = client.agents.create(model="claude-opus-4-6", system_prompt="設計専門")
coding_agent = client.agents.create(model="claude-sonnet-4-5", system_prompt="実装専門")
review_agent = client.agents.create(model="claude-haiku-4-5", system_prompt="レビュー専門")

# タスク実行
def build_feature(feature_description: str):
    design = design_agent.run(f"Design: {feature_description}")
    code = coding_agent.run(f"Implement: {design.output}")
    review = review_agent.run(f"Review: {code.output}")
    return {"design": design, "code": code, "review": review}
```

**効果:**
- **専門性向上**: 各エージェントが特定タスクに特化
- **コスト最適化**: Opus（高コスト）は設計のみ、Haiku（低コスト）はレビューに使用
- **並列実行**: サブエージェントを同時実行可能

## Model Driftの検出と対策

**Model Drift（モデル漂流）**とは、コンテキストが長くなると（100ステップ超）、LLMの指示追従能力が低下する現象です。症状として、最初に与えた指示を無視、同じエラーを繰り返す、無関係な情報を生成などがあります。

**対策:**
- **自動セッション再起動**: 50ステップごとにタスク理解度をチェックし、類似度50%未満でコンテキスト圧縮して新規セッション開始
- **ステップ制限**: max_steps=50-100で強制的に新規セッションに切り替え
- **定期的な再確認**: テストプロンプトを注入して初期タスクとの類似度を計算

**注意点:**
> Model Driftは100ステップ「前後」で発生するため、プロジェクトごとに最適な閾値をA/Bテストで決定してください。50-100ステップが一般的な目安です。
