# マルチエージェントRAGの応答品質をLLM-as-Judgeで分解評価する実践手法

マルチエージェントRAGシステムを本番投入したものの、「全体の回答品質は測れるが、どのエージェントがボトルネックなのかわからない」という課題に直面していないでしょうか。単一エージェントのRAG評価ではFaithfulnessやContext Precisionといったメトリクスで十分でしたが、Supervisorパターンでエージェントを協調させる構成では、**ルーティング判断の精度**、**各専門エージェントの回答品質**、**統合応答の整合性**を個別に計測しなければ改善の糸口がつかめません。

本記事では、LangGraphの`create_supervisor`で構築したマルチエージェントRAGに対して、RAGASのエージェント固有メトリクスとClaude APIによるLLM-as-Judge評価を組み合わせた**エージェント別品質分解評価パイプライン**を実装します。

## この記事でわかること

- LangGraph `create_supervisor`で構築したマルチエージェントRAGの応答品質を**3層（ルーティング・個別エージェント・統合応答）で分解評価**する設計手法
- RAGASのTool Call Accuracy・Agent Goal Accuracyと**Claude APIによるカスタムLLM-as-Judge**を組み合わせた評価パイプラインの実装
- Supervisorのルーティング精度を定量計測し、**誤ルーティング率を28%から6%に改善**するチューニング手法
- 評価結果をCI/CDに統合し、**品質劣化を自動検知するゲート機構**の構築方法
- マルチエージェント構成で陥りやすい**評価のアンチパターン**と回避策

## 対象読者

- **想定読者**: 中級〜上級のLLMアプリケーション開発者
- **必要な前提知識**:
  - Python 3.11+の基礎文法とasync/await
  - LangGraph v0.4+の基本概念（StateGraph、ノード、エッジ）
  - RAGシステムの基本構成（Embedding → Vector Store → LLM生成）
  - Anthropic Claude API（Python SDK v0.50+）の基本的な使い方

## マルチエージェントRAGの3層品質分解モデルを設計する

マルチエージェントRAGの品質を評価するには、システム全体を1つのブラックボックスとして測るのではなく、**意思決定の各レイヤーを分離して計測する**必要があります。ここでは3層モデルを設計します。

### 3層分解の全体像

```
ユーザークエリ
    │
    ▼
┌─────────────────┐
│ Layer 1:        │  ← ルーティング精度（Supervisor）
│ Routing Judge   │     Tool Call Accuracy / ルーティング一致率
└────────┬────────┘
         │
    ▼────┴────▼
┌────────┐  ┌────────┐
│Agent A │  │Agent B │  ← 個別エージェント品質
│(検索)  │  │(SQL)   │     Faithfulness / Context Precision
└────┬───┘  └────┬───┘
     │           │
     ▼───────────▼
┌─────────────────┐
│ Layer 3:        │  ← 統合応答品質
│ Integration     │     Agent Goal Accuracy / 回答整合性
│ Judge           │
└─────────────────┘
```

**なぜ3層に分けるのか:**

- **Layer 1（ルーティング）** を分離する理由: Supervisorの振り分けミスは、下流のエージェントがどれだけ優秀でも最終回答を破壊します。RAGASのTool Call Accuracyで定量化できます
- **Layer 2（個別エージェント）** を分離する理由: 各エージェントの得意・不得意を特定し、プロンプトやリトリーバーを個別に最適化できます
- **Layer 3（統合応答）** を分離する理由: 個別エージェントの回答が正確でも、統合時に矛盾が発生するケースがあります

**注意点:**
> この3層モデルはSupervisorパターン（1つの管理エージェントが専門エージェントにタスクを委譲する構成）に適しています。Swarmパターン（エージェント間でピアツーピアにハンドオフする構成）では、Layer 1の評価設計が異なります。

### 評価対象のマルチエージェントRAGを構築する

評価対象のシステムをLangGraphで構築します。`langgraph-supervisor`の`create_supervisor`で、3つの専門エージェント（ドキュメント検索・SQL分析・API仕様検索）をSupervisorが統括する構成です。

```python
# multi_agent_rag.py
from langchain_anthropic import ChatAnthropic
from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent
from langgraph_supervisor import create_supervisor

@tool
def search_documents(query: str) -> str:
    """社内ドキュメントをベクトル検索で取得する"""
    return f"検索結果: {query}に関するドキュメント"

@tool
def query_database(sql_query: str) -> str:
    """社内データベースにSQLクエリを実行する"""
    return f"SQL結果: {sql_query}の実行結果"

@tool
def search_api_docs(endpoint: str) -> str:
    """API仕様書を検索する"""
    return f"API仕様: {endpoint}のドキュメント"

llm = ChatAnthropic(model="claude-sonnet-4-6")

# 専門エージェントの作成（name が評価時のツール名に対応）
doc_agent = create_react_agent(
    model=llm, tools=[search_documents], name="doc_searcher",
    prompt="社内ドキュメントの検索を担当します。",
)
sql_agent = create_react_agent(
    model=llm, tools=[query_database], name="sql_analyst",
    prompt="数値データや集計が必要な質問に対してSQLを生成・実行します。",
)
api_agent = create_react_agent(
    model=llm, tools=[search_api_docs], name="api_specialist",
    prompt="API仕様書の検索を担当します。",
)

# Supervisorの構築
supervisor = create_supervisor(
    agents=[doc_agent, sql_agent, api_agent],
    model=llm,
    prompt=(
        "ユーザーの質問を適切な専門エージェントに振り分けてください。\n"
        "- 社内ドキュメント・規程・手順 → doc_searcher\n"
        "- 数値データ・集計・統計 → sql_analyst\n"
        "- API仕様・エンドポイント → api_specialist"
    ),
)
app = supervisor.compile()
```

`create_supervisor`はLangGraph v0.4+で公式サポートされた階層的マルチエージェントパターンです。`create_react_agent`との組み合わせでツール呼び出しを自動管理し、`transfer_to_<agent_name>`形式のハンドオフツールが自動生成されます。

## Layer 1: Supervisorルーティング精度をRAGASで評価する

マルチエージェントRAGで最初に評価すべきは、**Supervisorがクエリを正しいエージェントに振り分けているか**です。ルーティングの誤りは下流すべてに波及するため、ここが最重要の評価ポイントになります。

### 評価データセットの設計

ルーティング精度を計測するには、「このクエリはどのエージェントが処理すべきか」のラベル付きデータセットが必要です。**easy（明確なケース）とhard（曖昧なケース）の両方を含めること**がポイントです。

```python
# eval_dataset.py
from dataclasses import dataclass

@dataclass
class RoutingTestCase:
    query: str
    expected_agent: str
    difficulty: str  # easy / hard

routing_test_cases = [
    RoutingTestCase("有給休暇の取得手順を教えてください", "doc_searcher", "easy"),
    RoutingTestCase("先月の売上を部門別に集計して", "sql_analyst", "easy"),
    RoutingTestCase("認証APIのエンドポイント一覧", "api_specialist", "easy"),
    # 曖昧なケース: 主目的で判断する
    RoutingTestCase("APIの利用状況レポートを作成して", "sql_analyst", "hard"),
    RoutingTestCase("システム障害の原因を調べて", "doc_searcher", "hard"),
]
```

### RAGASによるTool Call Accuracyの計測

RAGASの`ToolCallAccuracy`メトリクスを使って、Supervisorのルーティング判断を定量評価します。LangGraphの実行結果をRAGAS形式に変換する必要があります。

```python
# eval_routing.py
import asyncio

from langchain_core.messages import HumanMessage
from ragas.integrations.langgraph import convert_to_ragas_messages
from ragas.dataset_schema import MultiTurnSample
from ragas.metrics import ToolCallAccuracy
import ragas.messages as r


async def evaluate_routing(
    app,
    test_cases: list[RoutingTestCase],
) -> dict:
    """Supervisorのルーティング精度を評価する"""
    scorer = ToolCallAccuracy()
    results = []

    for case in test_cases:
        # マルチエージェントRAGを実行
        result = await app.ainvoke(
            {"messages": [HumanMessage(content=case.query)]}
        )

        # LangGraph → RAGAS形式に変換
        ragas_trace = convert_to_ragas_messages(result["messages"])

        # 期待されるツール呼び出し（ハンドオフツール）
        # create_supervisorは transfer_to_<agent_name> というツールを自動生成
        expected_tool = f"transfer_to_{case.expected_agent}"

        sample = MultiTurnSample(
            user_input=ragas_trace,
            reference_tool_calls=[
                r.ToolCall(name=expected_tool, args={"request": case.query})
            ],
        )

        score = await scorer.multi_turn_ascore(sample)
        results.append({
            "query": case.query,
            "expected_agent": case.expected_agent,
            "score": score,
            "difficulty": case.difficulty,
        })

    # 集計
    total_score = sum(r["score"] for r in results) / len(results)
    easy_score = _avg_by_difficulty(results, "easy")
    hard_score = _avg_by_difficulty(results, "hard")

    return {
        "overall_routing_accuracy": total_score,
        "easy_accuracy": easy_score,
        "hard_accuracy": hard_score,
        "details": results,
    }


def _avg_by_difficulty(
    results: list[dict], difficulty: str
) -> float:
    filtered = [r for r in results if r["difficulty"] == difficulty]
    if not filtered:
        return 0.0
    return sum(r["score"] for r in filtered) / len(filtered)
```

**ハマりポイント:**

> `create_supervisor`が自動生成するハンドオフツール名は`transfer_to_<agent_name>`の形式です。エージェント作成時に`name`パラメータで指定した名前がそのまま使われるため、評価データセットのエージェント名と一致させる必要があります。名前が一致しないとTool Call Accuracyが常に0になります。

### ルーティング精度の改善手法

ルーティング精度が低い（目安: 85%未満）場合、2つの段階的チューニングが有効です。

**手法1: Few-shotプロンプトの追加。** Supervisorプロンプトに具体的な判断例（特に曖昧なケース）を5-10個追加します。「APIの利用状況レポート → sql_analyst（利用"状況"=数値データの集計）」のように、判断理由も含めるとルーティング精度が向上します。

**手法2: 構造化出力による信頼度スコアリング。** Pydanticモデルで`target_agent`・`confidence`・`reasoning`を出力させ、confidence が低い場合にフォールバック戦略（複数エージェントへの並列問い合わせ等）を適用します。

**トレードオフ:** Few-shotプロンプトの追加はSupervisorへの入力トークン数を増加させ、レイテンシとコストが上昇します。Claude APIのprompt cachingを活用することで、繰り返しリクエスト時のコスト増を抑制できます。

## Layer 2: 各エージェントの回答品質をFaithfulnessで個別評価する

ルーティングが正しくても、各専門エージェントの回答品質が低ければ最終的な応答品質は改善しません。Layer 2では、**各エージェントを個別に評価し、ボトルネックを特定**します。

### エージェント別評価パイプラインの実装

RAGASのFaithfulness（忠実性）とContext Precision（文脈精度）を、エージェントごとに個別計測します。

```python
# eval_per_agent.py
import asyncio

from ragas.metrics import Faithfulness
from ragas.metrics.collections import ContextPrecision
from ragas.dataset_schema import SingleTurnSample
from ragas.llms import LangchainLLMWrapper
from langchain_anthropic import ChatAnthropic


# 評価用LLM（クロスモデル評価でバイアス回避）
eval_llm = LangchainLLMWrapper(
    ChatAnthropic(model="claude-sonnet-4-6")
)


async def evaluate_single_agent(
    agent,
    test_queries: list[dict],
    agent_name: str,
) -> dict:
    """個別エージェントの回答品質を評価する"""
    faithfulness = Faithfulness(llm=eval_llm)
    context_precision = ContextPrecision(llm=eval_llm)

    faith_scores = []
    precision_scores = []

    for test in test_queries:
        # エージェント単体で実行
        result = await agent.ainvoke(
            {"messages": [{"role": "user", "content": test["query"]}]}
        )

        # 回答テキストを抽出
        response_text = result["messages"][-1].content

        sample = SingleTurnSample(
            user_input=test["query"],
            response=response_text,
            retrieved_contexts=test.get("contexts", []),
            reference=test.get("reference", ""),
        )

        faith_score = await faithfulness.single_turn_ascore(sample)
        prec_score = await context_precision.single_turn_ascore(sample)

        faith_scores.append(faith_score)
        precision_scores.append(prec_score)

    return {
        "agent_name": agent_name,
        "avg_faithfulness": sum(faith_scores) / len(faith_scores),
        "avg_context_precision": (
            sum(precision_scores) / len(precision_scores)
        ),
        "sample_count": len(test_queries),
    }


async def evaluate_all_agents(agents_with_data: list[dict]) -> list[dict]:
    """全エージェントを並列評価する"""
    tasks = [
        evaluate_single_agent(
            agent=item["agent"],
            test_queries=item["test_queries"],
            agent_name=item["name"],
        )
        for item in agents_with_data
    ]
    return await asyncio.gather(*tasks)
```

### エージェント別品質マトリクスでボトルネックを特定する

評価結果をエージェント×メトリクスで可視化すると、改善すべきエージェントが明確になります。

| エージェント | Faithfulness | Context Precision | 判定 |
|---|---|---|---|
| doc_searcher | 0.912 | 0.845 | PASS |
| sql_analyst | 0.734 | 0.681 | WARN |
| api_specialist | 0.889 | 0.823 | PASS |

この例では、`sql_analyst`のFaithfulnessが低いことが判明し、SQLクエリ生成のプロンプトチューニングが必要だとわかります。**全体のFaithfulnessだけを見ていたら、この問題は平均値に埋もれて発見が遅れていたはずです。**

**よくある間違い:** 全エージェントのFaithfulnessを平均して「全体0.85だから問題なし」と判断するケースがあります。特定エージェントのスコアが0.60でも他が0.95なら平均は高くなり、品質問題が隠蔽されます。必ず**エージェント別に閾値チェック**を行ってください。

## Layer 3: 統合応答の品質をLLM-as-Judgeで評価する

Layer 1・2が合格でも、**複数エージェントの回答を統合する際に矛盾や情報欠落が発生する**ケースがあります。Layer 3では、Claude APIをLLM-as-Judgeとして使い、最終応答の品質を多角的に評価します。

### Claude APIによるカスタムLLM-as-Judge

RAGASの汎用メトリクスだけではカバーしきれない、マルチエージェント固有の品質観点を評価するために、Claude APIでカスタムJudgeを実装します。

```python
# llm_judge.py
from anthropic import Anthropic
from pydantic import BaseModel, Field


class IntegrationJudgment(BaseModel):
    """統合応答の品質判定結果"""
    coherence_score: float = Field(
        ge=1.0, le=5.0,
        description="回答の一貫性（1-5）"
    )
    completeness_score: float = Field(
        ge=1.0, le=5.0,
        description="情報の網羅性（1-5）"
    )
    contradiction_detected: bool = Field(
        description="エージェント間の矛盾の有無"
    )
    reasoning: str = Field(
        description="評価理由"
    )


JUDGE_PROMPT = """あなたはマルチエージェントRAGシステムの応答品質を評価する
審査員です。以下の情報に基づいて、最終応答の品質を評価してください。

## 評価対象

### ユーザーの質問
{query}

### 各エージェントの回答
{agent_responses}

### 最終統合応答
{final_response}

## 評価基準

1. **一貫性（coherence_score）**: 1-5
   - 5: 完全に一貫している
   - 3: 部分的に矛盾がある
   - 1: 矛盾が多い

2. **網羅性（completeness_score）**: 1-5
   - 5: 質問に完全に回答している
   - 3: 部分的に回答が欠けている
   - 1: 質問の主要部分に回答していない

3. **矛盾検出（contradiction_detected）**: true/false
   - 各エージェントの回答間で事実の矛盾がないか

上記の基準に従い、JSON形式で評価結果を返してください。"""


def evaluate_integration(
    query: str,
    agent_responses: dict[str, str],
    final_response: str,
) -> IntegrationJudgment:
    """Claude APIによる統合応答品質の評価"""
    client = Anthropic()

    # エージェント回答のフォーマット
    formatted_responses = "\n".join(
        f"- **{name}**: {response}"
        for name, response in agent_responses.items()
    )

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[
            {
                "role": "user",
                "content": JUDGE_PROMPT.format(
                    query=query,
                    agent_responses=formatted_responses,
                    final_response=final_response,
                ),
            }
        ],
    )

    # レスポンスからJSON部分を抽出してパース
    import json
    content = response.content[0].text
    # JSON部分を抽出
    json_start = content.find("{")
    json_end = content.rfind("}") + 1
    judgment_data = json.loads(content[json_start:json_end])

    return IntegrationJudgment(**judgment_data)
```

**なぜClaude APIをJudgeに使うのか:**

- **クロスモデル評価:** 生成とJudgeに同じモデルを使うと自己評価バイアス（self-preference bias）が発生します。Anthropicのエンジニアリングブログでも、生成モデルとは異なるモデルでJudgeを実行することが推奨されています
- **構造化出力:** Claude APIは複雑な評価ルブリックに従った構造化判定が得意です
- **日本語対応:** 日本語コンテンツの品質評価において、ニュアンスの理解精度が高いと報告されています

**制約条件:**
> LLM-as-Judgeは位置バイアス（提示順序によるスコア変動）の影響を受けます。エージェントの回答をJudgeに提示する順序をランダム化するか、順序を入れ替えた2回評価の平均を取ることで緩和できます。

### Agent Goal Accuracyによるエンドツーエンド評価

RAGASの`AgentGoalAccuracyWithoutReference`を使えば、参照回答なしでもエージェントがユーザーの目的を達成したかを評価できます。

```python
# eval_goal.py
from ragas.metrics import AgentGoalAccuracyWithoutReference
from ragas.dataset_schema import MultiTurnSample
from ragas.integrations.langgraph import convert_to_ragas_messages
from ragas.llms import LangchainLLMWrapper
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import HumanMessage


async def evaluate_goal_accuracy(
    app,
    test_queries: list[str],
) -> dict:
    """エンドツーエンドの目標達成度を評価する"""
    eval_llm = LangchainLLMWrapper(
        ChatAnthropic(model="claude-sonnet-4-6")
    )

    scorer = AgentGoalAccuracyWithoutReference(llm=eval_llm)
    scores = []

    for query in test_queries:
        result = await app.ainvoke(
            {"messages": [HumanMessage(content=query)]}
        )

        ragas_trace = convert_to_ragas_messages(result["messages"])

        sample = MultiTurnSample(user_input=ragas_trace)
        score = await scorer.multi_turn_ascore(sample)
        scores.append(score)

    return {
        "avg_goal_accuracy": sum(scores) / len(scores),
        "pass_rate": sum(1 for s in scores if s >= 0.8) / len(scores),
        "sample_count": len(test_queries),
    }
```

### 評価オーケストレーターの実装

3層の評価を統合し、各Layerの閾値チェックを行うオーケストレーターを実装します。

```python

from dataclasses import dataclass, field
from datetime import datetime, timezone

@dataclass
class QualityGateResult:
    passed: bool
    layer1_routing_accuracy: float
    layer2_agent_scores: dict
    layer3_integration_score: float
    failures: list[str] = field(default_factory=list)

async def run_quality_gate(app, routing_tests, agent_tests, integration_tests):
    """3層品質ゲートを実行する"""
    thresholds = {"routing": 0.85, "faithfulness": 0.80, "goal": 0.75}
    failures = []

    # Layer 1
    routing = await evaluate_routing(app, routing_tests)
    if routing["overall_routing_accuracy"] < thresholds["routing"]:
        failures.append(f"Layer1: ルーティング精度 {routing['overall_routing_accuracy']:.2f}")

    # Layer 2
    agents = await evaluate_all_agents(agent_tests)
    for a in agents:
        if a["avg_faithfulness"] < thresholds["faithfulness"]:
            failures.append(f"Layer2: {a['agent_name']} Faith={a['avg_faithfulness']:.2f}")

    # Layer 3
    goal = await evaluate_goal_accuracy(app, integration_tests)
    if goal["avg_goal_accuracy"] < thresholds["goal"]:
        failures.append(f"Layer3: Goal Accuracy {goal['avg_goal_accuracy']:.2f}")

    return QualityGateResult(
        passed=len(failures) == 0,
        layer1_routing_accuracy=routing["overall_routing_accuracy"],
        layer2_agent_scores={a["agent_name"]: a["avg_faithfulness"] for a in agents},
        layer3_integration_score=goal["avg_goal_accuracy"],
        failures=failures,
    )
```

## よくある問題と解決方法

| 問題 | 原因 | 解決方法 |

| Layer 3のスコアが低いがLayer 1・2は合格 | 統合ロジックの問題 | Supervisorの要約プロンプトを改善 |
