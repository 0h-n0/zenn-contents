## AIエージェント評価の3層構造

AIエージェントの評価は、システムの複雑さに応じて**プロンプトレベル・RAGレベル・エージェントレベルの3層**に分類されます。各層で異なる指標を適用することで、包括的な品質管理が実現します。

### プロンプトレベル評価

単一ターンのLLM呼び出しを対象とした基礎的な評価層です。

**主要指標**:
- **出力形式の正確性**: JSON/XML等の構造化データが正しく生成されるか
- **トーン・スタイルの一貫性**: ビジネス文書・カジュアルな対話等の指定遵守率
- **命令遵守率**: プロンプトで指定した制約条件の充足度

**実装例（Python + Pydantic）**:

```python
from pydantic import BaseModel, Field
from openai import OpenAI

class StructuredOutput(BaseModel):
    """出力形式を厳密に定義"""
    summary: str = Field(description="要約（100文字以内）")
    sentiment: str = Field(description="ポジティブ/ネガティブ/中立")
    confidence: float = Field(ge=0.0, le=1.0, description="信頼度")

client = OpenAI()

def evaluate_prompt_level(prompt: str, expected_format: type[BaseModel]):
    """プロンプトレベル評価"""
    response = client.beta.chat.completions.parse(
        model="gpt-4o-2024-08-06",
        messages=[{"role": "user", "content": prompt}],
        response_format=expected_format,
    )

    # 1. 出力形式の検証（Pydanticが自動実施）
    parsed = response.choices[0].message.parsed

    # 2. トークン使用量とコストの測定
    usage = response.usage
    cost = (usage.prompt_tokens * 0.0025 + usage.completion_tokens * 0.01) / 1000

    # 3. 評価結果を記録
    return {
        "format_valid": True,  # Pydantic検証成功
        "token_count": usage.total_tokens,
        "cost_usd": cost,
        "latency_ms": response.response_time_ms,
    }
```

**なぜこの実装か**:
- Pydanticの型検証により出力形式の正確性を保証
- トークン使用量とコストを自動測定し、経済性を可視化
- 失敗時は `ValidationError` で原因を特定可能

**注意点**:
> プロンプトレベル評価は単一ターン限定です。マルチターン会話やツール呼び出しを含むケースでは、次のRAGレベル・エージェントレベル評価が必要です。

### RAGレベル評価

検索拡張生成（RAG）システムでは、**検索品質**と**生成精度**を独立して評価します。

**検索フェーズの指標**:

| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| **Precision@K** | 0.8以上 | 上位K件中の関連文書割合 |
| **Recall@K** | 0.7以上 | 全関連文書中の検索成功率 |
| **NDCG@K** | 0.75以上 | ランキング品質（正規化DCG） |
| **Context Relevance** | 0.85以上 | 検索チャンクが質問に関連する度合い |

**生成フェーズの指標**:

| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| **Answer Grounding** | 0.9以上 | 応答が検索文書に根拠を持つ割合 |
| **Hallucination Rate** | 5%未満 | 検索文書に存在しない情報の生成率 |
| **Completeness** | 0.85以上 | 質問に対する網羅性 |

**実装例（Ragas + LangChain）**:

```python
from ragas import evaluate, EvaluationDataset
from ragas.metrics import (
    Faithfulness,
    ResponseRelevancy,
    LLMContextRecall,
)
from ragas.metrics.collections import ContextPrecision

# RAG評価データセット（v0.2+のフィールド名）
eval_dataset = EvaluationDataset.from_list([{
    "user_input": "LLMの評価指標は？",
    "response": "プロンプトレベル、RAGレベル、エージェントレベルの3層です",
    "retrieved_contexts": ["LLM評価は3層構造で実施される。プロンプト評価、RAG評価、エージェント評価。"],
    "reference": "3層構造: プロンプト、RAG、エージェント",
}])

# Ragasで自動評価（v0.2+ API）
result = evaluate(
    dataset=eval_dataset,
    metrics=[
        ContextPrecision(),     # 検索精度
        LLMContextRecall(),     # 検索網羅性
        Faithfulness(),         # 生成の忠実性（幻覚検出）
        ResponseRelevancy(),    # 応答の関連性
    ],
)

print(f"Context Precision: {result['context_precision']:.2f}")
print(f"Faithfulness: {result['faithfulness']:.2f}")
```

**なぜRagasを選ぶか**:
- LLM-as-judgeを活用し、人手評価コストを削減
- 検索と生成を独立評価し、ボトルネックを特定可能
- LangChain・LlamaIndexと統合可能

**落とし穴**:
> Ragasの評価スコアは評価用LLMの品質に依存します。GPT-4o等の高性能モデルを評価器として使用し、定期的に人間評価で較正してください。

### エージェントレベル評価

複数のLLM呼び出し、ツール使用、マルチターン状態を持つエージェントでは、**タスク達成能力**と**推論品質**を評価します。

**主要指標**:

| 指標 | 定義 | 目標値 |
|------|------|--------|
| **Task Success Rate** | タスク完了の成功割合 | 95%以上 |
| **Tool Selection Accuracy** | 適切なツールを選択した割合 | 90%以上 |
| **Reasoning Quality** | 推論チェーンの論理性（LLM-as-judge） | 0.85以上 |
| **Error Recovery Rate** | エラー発生時の回復成功率 | 80%以上 |
| **Average Turns to Completion** | タスク完了までの平均ターン数 | 3ターン以下 |

**実装例（LangSmith評価）**:

```python
from langsmith import Client, evaluate
from openai import OpenAI

client = Client()
openai_client = OpenAI()

# エージェント評価関数（v2 signature）
def task_success_evaluator(outputs: dict, reference_outputs: dict) -> dict:
    """タスク成功判定"""
    agent_output = outputs.get("output", "")
    expected = reference_outputs.get("expected_result", "")

    # LLM-as-judgeで成功判定（OpenAIクライアントを使用）
    judge_prompt = f"""
    Agent Output: {agent_output}
    Expected Result: {expected}

    Did the agent successfully complete the task? Answer 'yes' or 'no'.
    """

    judge_response = openai_client.chat.completions.create(
        model="gpt-4o-2024-08-06",
        messages=[{"role": "user", "content": judge_prompt}],
    )

    success = judge_response.choices[0].message.content.lower() == "yes"
    return {"key": "task_success", "score": 1.0 if success else 0.0}

# エージェントを評価
results = evaluate(
    target=my_agent_function,  # 評価対象のエージェント
    data="agent-test-dataset",  # テストデータセット名
    evaluators=[task_success_evaluator],
    experiment_prefix="agent-eval-v1",
)

print(f"Task Success Rate: {results['task_success_evaluator']['mean']:.1%}")
```

**なぜLangSmithか**:
- LangChain統合でマルチターン会話を自動追跡
- 各ターンでのツール呼び出しと推論ステップを可視化
- CI/CDパイプラインに統合可能

**実務での課題**:
> エージェント評価の最大の課題は「タスク成功の定義」です。曖昧な目標（「ユーザーを満足させる」）ではなく、測定可能な基準（「3ターン以内に正確な回答を提供」）を設定してください。

## ビジネス指標とシステム指標の統合

Microsoftが提唱する評価フレームワークでは、**Understanding（理解）、Reasoning（推論）、Response Quality（応答品質）**の3軸でビジネス成果とシステム性能を統合します。

### 3軸評価モデル

#### 1. Understanding（理解）

**ビジネス指標**:
- 初回理解成功率（First Intent Resolution）: 85%以上
- ユーザー再質問率: 15%未満

**システム指標**:
- エンティティ抽出精度: 0.9以上
- インテント分類精度: 0.92以上

**測定方法**:

```python
def evaluate_understanding(user_query: str, agent_interpretation: dict):
    """理解精度の評価"""
    # 1. エンティティ抽出の正確性
    expected_entities = {"product": "iPhone 15", "action": "購入"}
    extracted = agent_interpretation.get("entities", {})
    entity_accuracy = len(set(expected_entities) & set(extracted)) / len(expected_entities)

    # 2. インテント分類の正確性
    expected_intent = "purchase_intent"
    predicted_intent = agent_interpretation.get("intent")
    intent_correct = expected_intent == predicted_intent

    return {
        "entity_accuracy": entity_accuracy,
        "intent_correct": int(intent_correct),
        "understanding_score": (entity_accuracy + int(intent_correct)) / 2,
    }
```

#### 2. Reasoning（推論）

**ビジネス指標**:
- 論理的一貫性: 人間評価で4.0/5.0以上
- 問題解決効率: 平均3ステップ以内

**システム指標**:
- 推論チェーンの完全性: 全ステップが記録されているか
- ツール選択の最適性: 不要なツール呼び出しが無いか

**トレーシングによる可視化**:

```python
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import ConsoleSpanExporter, SimpleSpanProcessor

# OpenTelemetryでトレーシング設定
provider = TracerProvider()
processor = SimpleSpanProcessor(ConsoleSpanExporter())
provider.add_span_processor(processor)
trace.set_tracer_provider(provider)

tracer = trace.get_tracer(__name__)

def agent_reasoning_step(step_name: str, context: dict):
    """推論ステップをトレース"""
    with tracer.start_as_current_span(step_name) as span:
        span.set_attribute("reasoning.context", str(context))
        span.set_attribute("reasoning.step_number", context.get("step", 0))

        # 推論実行
        result = execute_reasoning_logic(context)

        span.set_attribute("reasoning.result", str(result))
        span.set_attribute("reasoning.success", result.get("success", False))

        return result

# 推論チェーンの実行
with tracer.start_as_current_span("agent_task") as root_span:
    root_span.set_attribute("task.id", "task-12345")

    step1 = agent_reasoning_step("analyze_query", {"query": "..."})
    step2 = agent_reasoning_step("select_tool", {"analysis": step1})
    step3 = agent_reasoning_step("execute_tool", {"tool": step2})
```

**トレーシングの利点**:
- 各推論ステップの実行時間を測定し、ボトルネックを特定
- エラー発生時の根本原因を特定（どのステップで失敗したか）
- 分散システムでのエンドツーエンドの処理フローを可視化

#### 3. Response Quality（応答品質）

**ビジネス指標**:
- 顧客満足度（CSAT）: 4.5/5.0以上
- 初回解決率（FCR）: 85%以上

**システム指標**:
- 応答の完全性: 質問の全要素に答えているか（0.9以上）
- 応答の簡潔性: 不要な情報が含まれていないか（0.85以上）
- 安全性: 有害コンテンツ検出率（0%目標）

**ガードレール実装例**:

```python

from guardrails.hub import ValidLength, ToxicLanguage

# エージェント応答を検証

if validated_response.validation_passed:
    print("応答品質OK")
else:
    print(f"検証失敗: {validated_response.error}")
```

## 可観測性と評価の統合実装

LangChainの調査で明らかになった「可観測性89% vs 評価52%」のギャップを埋めるには、**Observability-by-Design**の原則で最初から評価を組み込みます。

### 実装パターン1: トレーシング駆動評価

**4つのコア要素**:

1. **トレーシング**: 各エージェント実行の全ステップを記録
2. **ログ**: 構造化されたJSON形式でイベントを記録
3. **メトリクス**: 数値指標（レイテンシ、トークン数、コスト）を集計
4. **評価**: 自動スコアリング（LLM-as-judge + ヒューリスティック）

**実装例（Braintrust統合）**:

```python
from braintrust import init_logger, current_span

# Braintrustロガー初期化
logger = init_logger(project="agent-evaluation", api_key="YOUR_API_KEY")

def agent_with_observability(user_query: str):
    """可観測性を組み込んだエージェント実行"""
    with logger.start_span(name="agent_task", input={"query": user_query}) as span:
        # 1. トレーシング: クエリ分析
        with logger.start_span(name="analyze_query") as analysis_span:
            analysis = analyze_user_query(user_query)
            analysis_span.log(output=analysis)

        # 2. メトリクス: トークン使用量を記録
        span.log(metrics={"tokens_used": 150, "cost_usd": 0.0045})

        # 3. ツール実行をトレース
        with logger.start_span(name="tool_execution") as tool_span:
            tool_result = execute_tool(analysis["selected_tool"])
            tool_span.log(output=tool_result)

        # 4. 最終応答生成
        response = generate_response(analysis, tool_result)

        # 5. 自動評価をトレースに付与
        with logger.start_span(name="evaluation") as eval_span:
            eval_result = evaluate_response(user_query, response)
            eval_span.log(
                scores={
                    "task_success": eval_result["success"],
                    "response_quality": eval_result["quality_score"],
                }
            )

        span.log(output={"response": response})
        return response

# 実行
result = agent_with_observability("最新のiPhoneを購入したい")
```

**このパターンの利点**:
- トレースと評価を同じフレームワークで管理
- 本番環境のトレースを自動的にテストセットに変換
- CI/CDで同じ評価基準を適用可能

```python
import json
from pathlib import Path
from langsmith import Client, evaluate

def load_test_cases(dataset_path: str):
    """テストケース読み込み"""
    return json.loads(Path(dataset_path).read_text())

def run_evaluation(dataset_path: str, baseline_threshold: float):
    """エージェント評価を実行"""
    client = Client()

    # LangSmithでオフライン評価

        target=my_production_agent,
        data=dataset_path,
```

### 実装パターン3: OpenTelemetry標準化

**ベンダーロックイン回避**:

```python
from opentelemetry import trace, metrics
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.exporter.otlp.proto.grpc.metric_exporter import OTLPMetricExporter
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.metrics.export import PeriodicExportingMetricReader

# OTLPエクスポーター設定（Datadog/Grafana/Jaeger互換）
trace_provider = TracerProvider()
trace_provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="http://localhost:4317"))

trace.set_tracer_provider(trace_provider)

# メトリクスプロバイダー
metric_reader = PeriodicExportingMetricReader(
    OTLPMetricExporter(endpoint="http://localhost:4317"),
    export_interval_millis=5000,

# トレーサーとメーターを取得
tracer = trace.get_tracer("agent.tracer")
meter = metrics.get_meter("agent.meter")

# メトリクス定義
task_success_counter = meter.create_counter(
    name="agent.task.success",
```
