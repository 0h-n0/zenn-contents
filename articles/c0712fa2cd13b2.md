# Malleable Softwareとは何か——LLM時代の動的ソフトウェアとその実現要件

## この記事でわかること

- **Malleable Software（可鍛性ソフトウェア）** の概念と、従来のソフトウェア開発との違い
- LLM・コーディングエージェントが「ユーザー主導のコード修正」を可能にする仕組み
- パーソナルソフトウェア（Software for One）の具体的なユースケースと成功事例
- この考え方のメリット・デメリット・トレードオフの整理
- Malleable Softwareを実現するために必要な技術要素と設計原則

## 対象読者

- **想定読者**: 中級者〜上級者のソフトウェアエンジニア・プロダクトマネージャー
- **必要な前提知識**:
  - LLM（Large Language Model）の基本的な理解
  - プログラミングの基礎（言語は問わない）
  - ソフトウェア開発プロセスの概要

## 結論・成果

2026年現在、LLMとコーディングエージェントの進化により、**ソフトウェアをユーザー自身が修正・カスタマイズできる世界**が現実味を帯びてきています。Anthropicのレポートによると、開発者の92%がAIコーディングツールを日常使用し、全コードの41%がAI生成という状況です。一方で、AI生成コードの45%がセキュリティテストに不合格というVeracodeの2025年報告もあり、Malleable Softwareの実現には技術的・社会的な課題が残ります。

本記事では、Ink & SwitchのMalleable Softwareエッセイ、Geoffrey Littの研究、Andrej KarpathyのVibe Coding概念を軸に、動的ソフトウェアとパーソナルソフトウェアの考え方を整理します。

## Malleable Softwareの概念を理解する

Malleable Software（可鍛性ソフトウェア）は、Ink & Switchの研究グループが2025年に発表したエッセイで体系化された概念です。パーソナルコンピューティングの原点——ユーザーが自由に形を変えられる「新しい粘土」——を、現代のロックダウンされたアプリの世界で取り戻そうという提案です。

### 従来のソフトウェアとの違い

従来のソフトウェア開発では、ユーザーの要求は開発チームを経由して実装されます。このモデルには明確な限界があります。

| 観点 | 従来型ソフトウェア | Malleable Software |
|------|--------------------|--------------------|
| カスタマイズ主体 | 開発チーム | エンドユーザー |
| 変更のタイミング | リリースサイクルに依存 | 使用時にリアルタイム |
| 必要スキル | プログラミング必須 | 自然言語で指示可能 |
| 変更の粒度 | 設定・プラグインの範囲 | コードレベルの修正 |
| フィードバックループ | 数日〜数ヶ月 | 数秒〜数分 |

Malleable Softwareの本質は、「修正が例外的な行為ではなく、日常的な行為になる」ことにあります。

### Ink & Switchの3つの設計パターン

Ink & Switchのエッセイでは、Malleable Softwareを実現するための3つの設計パターンが提示されています。

**1. Gentle Slope（緩やかな坂道）**: ユーザーから作り手へ

ExcelやHyperCardのように、使い始めはシンプルに、段階的にカスタマイズの深度を増していくアプローチです。MacLeanらのフレームワークに基づき、カスタマイズは「断崖」ではなく「坂道」であるべきとされています。

```python
# Gentle Slopeの段階例
# Level 1: 設定変更（GUI操作のみ）
config = {"theme": "dark", "font_size": 14}

# Level 2: 数式・フィルタ（Excelライクな操作）
# ユーザーが自然言語で「売上が100万以上のものだけ表示」と指示
filter_rule = "sales > 1_000_000"

# Level 3: スクリプト修正（LLMアシスト）
# ユーザー: 「集計ロジックを月次から週次に変更して」
# LLMが既存コードを解析し、修正案を提示
def aggregate_data(df, period="weekly"):
    """集計期間を変更可能にした関数"""
    if period == "weekly":
        return df.resample("W").sum()
    elif period == "monthly":
        return df.resample("M").sum()
    return df
```

**2. Tools over Applications（アプリではなくツール）**

Ink & Switchは「キッチンの比喩」を用いています。専用ガジェット（=アプリ）は1つの問題しか解けませんが、汎用ツール（=包丁）は多くのタスクに対応できます。重要なのは、**共有データ**と**UIの合成**です。

**3. Communal Creation（コミュニティによる共同創作）**

個人のカスタマイズを超え、コミュニティ全体で改変・共有していくモデルです。「ローカルデベロッパー」と呼ばれる、その組織に詳しいカスタマイズ担当者が、他のメンバーのための修正を行います。

> **注意**: これらの設計パターンは理想的なモデルであり、2026年現在のツールで完全に実現されているわけではありません。既存のプラットフォームはいずれも部分的な実装に留まっています。

## Vibe CodingとSoftware for Oneを実践する

2025年2月、OpenAI共同創業者のAndrej Karpathyが提唱した**Vibe Coding**は、Malleable Softwareの一形態として大きな注目を集めました。LLMに自然言語で指示し、コードを生成させるアプローチです。

### Vibe Codingの現在地

2026年の統計によると、開発者の92%がAIコーディングツールを日常使用し、82%がAIコーディングツールを少なくとも週次で使用しています（Stack Overflow 2025年開発者調査）。

ニューヨーク・タイムズのジャーナリストKevin Rooseは、プログラミング未経験の立場から複数のアプリケーションをVibe Codingで作成し、これを**「Software for One（一人のためのソフトウェア）」**と呼びました。LunchBox Buddyという日常ツールを自作した事例は、非開発者がパーソナルソフトウェアを構築できる可能性を示しています。

### パーソナルソフトウェアの具体的ユースケース

LLMを活用して「自分だけのソフトウェア」を作成するユースケースは以下の通りです。

```typescript
// ユースケース1: ワンオフスクリプト
// 「自分のSlackメッセージを分析して、最も活発な時間帯を可視化」

// ユースケース2: 既存ツールのカスタマイズ
// 「このダッシュボードに自分のKPIだけを表示するフィルタを追加」

// ユースケース3: 業務プロセスの自動化
// 「毎週月曜のレポート作成を自動化するスクリプト」

// 実装例: Claude Codeでの対話的なカスタマイズ
// ユーザーが自然言語で依頼 → LLMがコードを生成・修正

interface PersonalTool {
  name: string;
  description: string;
  createdBy: "end-user";       // 非開発者が作成
  modifiedWith: "llm-assist";  // LLMアシストで修正
  scope: "personal" | "team";  // 利用範囲
}

// 例: 個人用のタスク管理ツール
const myTaskManager: PersonalTool = {
  name: "MyTaskFlow",
  description: "自分のワークフローに最適化されたタスク管理",
  createdBy: "end-user",
  modifiedWith: "llm-assist",
  scope: "personal",
};
```

**なぜこのアプローチが注目されるのか:**

- 既存のSaaSツールは「最大公約数的な設計」であり、個々のニーズに完全にフィットしない
- LLMによるコード生成・修正のコストが大幅に低下し、1人のためのソフトウェアが経済的に成立するようになった
- Y Combinator 2025年冬バッチの25%が、コードの95%以上をAI生成したスタートアップだったと報告されている

## メリットとデメリットを分析する

Malleable SoftwareとパーソナルソフトウェアにはApproachとしての明確なメリットとデメリットがあります。ここでは客観的なデータに基づいて整理します。

### メリット

**1. ソフトウェアの民主化**

プログラミング知識がなくても、LLMを通じてソフトウェアを作成・修正できます。Microsoftの報告によると、セキュリティチームがコード分析に、リサーチチームがデータ可視化フロントエンドの構築に、非技術職がネットワーク問題のデバッグにAIコーディングツールを活用している事例があります。

**2. フィードバックループの短縮**

従来の「要件定義 → 設計 → 実装 → テスト → リリース」のサイクルが、「自然言語で指示 → 即座にコード生成 → 動作確認」に短縮されます。Vibe Codingでは**プロジェクトの80%を数時間で完成**できるとされています。

**3. コンテキスト固有の最適化**

汎用SaaSでは対応できない、特定の業務フロー・個人の習慣に合わせた最適化が可能です。Ink & Switchのエッセイでは、医療従事者がEHR（電子健康記録）システムを専門科のワークフローに合わせてカスタマイズできれば、**記録されている燃え尽き症候群の軽減**に寄与する可能性が指摘されています。

**4. 「Situated Software」の実現**

Clay Shirkyが提唱した概念で、特定のコンテキスト向けに設計されたソフトウェアは、マスマーケット向けの設計よりも実装が容易であるという考え方です。Malleable Softwareはこの延長線上にあります。

### デメリット

**1. セキュリティリスク**

Veracodeの2025年報告によると、AI生成コードサンプルの約45%がセキュリティテストに不合格で、OWASP Top 10に含まれる脆弱性を含んでいました。2025年5月には、Vibe CodingプラットフォームLovableで生成された1,645のWebアプリのうち170件で個人情報が公開アクセス可能な状態にあったと報告されています。

```python
# セキュリティリスクの例: AI生成コードによくある問題
# 悪い例（AI生成コードに頻出するパターン）
import sqlite3

def get_user(user_id: str) -> dict:
    conn = sqlite3.connect("app.db")
    # SQLインジェクション脆弱性
    cursor = conn.execute(f"SELECT * FROM users WHERE id = '{user_id}'")
    return dict(cursor.fetchone())

# 安全な実装（パラメータ化クエリ）
def get_user_safe(user_id: str) -> dict:
    conn = sqlite3.connect("app.db")
    cursor = conn.execute(
        "SELECT * FROM users WHERE id = ?",
        (user_id,)  # パラメータバインディング
    )
    return dict(cursor.fetchone())
```

**2. 保守性の問題**

CodeRabbitの2025年12月の分析（GitHubの470のオープンソースPR対象）によると、AI共著コードは人間のコードと比較して**ロジックエラーが1.7倍**、**セキュリティ脆弱性が2.74倍**高い割合で含まれていました。これは「Vibe Codingハングオーバー」と呼ばれ、3ヶ月後に誰もコードの動作を説明できなくなる問題です。

Fast Companyの報告では、2027年までにAI生成コードの構造的問題から**1.5兆ドルの技術的負債**が蓄積されると予測されています。

**3. 生産性パラドックス**

2025年7月のMETR（AIモデル評価機関）の調査では、**経験豊富なオープンソース開発者がAIコーディングツール使用時に19%遅くなる**という結果が報告されています。開発者自身は「24%速くなった」と予測し、使用後も「20%速くなった」と感じていたにもかかわらずです。

**4. 品質の不均一性**

Vibe Codingでは「最初の80%は数時間で完成するが、プロダクション品質にするための残り20%」に専門知識が必要です。この「ラストマイル問題」は、非開発者がパーソナルソフトウェアを構築する際の大きな障壁です。

| カテゴリ | メリット | リスク/課題 |
|----------|----------|-------------|
| アクセシビリティ | 非開発者がソフトウェアを作成可能 | セキュリティ脆弱性の増加（45%不合格率） |
| 開発速度 | 80%が数時間で完成 | 残り20%の品質確保に専門知識が必要 |
| カスタマイズ性 | 個人・組織固有の最適化 | 保守性低下（1.7倍のロジックエラー） |
| コスト | 1人用ソフトウェアが経済的に成立 | 技術的負債の蓄積（推定1.5兆ドル/2027年） |

## 実現に必要な技術要素と設計原則を整理する

Malleable Softwareを実現するには、LLMの進化だけでは不十分です。複数の技術要素と設計原則が必要になります。

### 必要な技術要素

**1. コードの理解と修正を可能にするLLM基盤**

2026年現在、Claude Code、Cursor、GitHub Copilotなどのツールがリポジトリ全体を理解し、マルチファイルの変更を行い、テストを実行して反復する能力を持っています。Anthropicの2026年Agentic Coding Trends Reportでは、開発者がAIを作業の60%に統合し、委任タスクの80-100%で積極的な監視を維持していると報告されています。

```python
# LLM基盤の活用パターン: ユーザーの修正依頼 → コード変更
# 実際のClaude Code使用例のイメージ

# ユーザー: 「このグラフの色をブランドカラーに変更して」
# LLMが既存コード解析 → 修正案を生成 → ユーザーが承認

# Before（LLM修正前）
CHART_CONFIG = {
    "primary_color": "#3498db",   # デフォルトの青
    "secondary_color": "#2ecc71", # デフォルトの緑
}

# After（LLM修正後）
CHART_CONFIG = {
    "primary_color": "#FF6B35",   # ブランドオレンジ
    "secondary_color": "#004E89", # ブランドネイビー
}
# LLMが関連するCSS/テーマファイルも同時に更新
```

**2. 標準化されたプロトコル**

AnthropicのModel Context Protocol（MCP）やGoogleのAgent-to-Agent Protocol（A2A）は、エージェントが外部ツール・データベース・APIに接続するための標準を確立しつつあります。MCPは2025年を通じて広く採用が進みました。

**3. ローカルファーストなデータアーキテクチャ**

Ink & Switchの研究プロトタイプでは、以下の技術が検証されています。

- **Automerge**: ローカルファーストの協調編集ライブラリ
- **Cambria**: 異なるツール間でのスキーマ変換をリアルタイムに行うシステム
- **PushPin**: ドキュメントとUIを分離し、同じデータに対して異なるエディタを使用可能にするアーキテクチャ

**4. サンドボックス実行環境**

ユーザーが修正したコードを安全に実行するための隔離環境が必要です。WebAssembly（Wasm）ベースのサンドボックスや、コンテナベースの実行環境が候補になります。

### 設計原則

Geoffrey Littの研究から導かれる設計原則を以下に整理します。

```
1. Direct Manipulation First（直接操作を優先）
   - チャットUIだけでなく、直接操作可能なUIを提供
   - 「内側のループ（即時フィードバック）」と
     「外側のループ（LLMへの意識的な依頼）」を分離

2. Gradual Engagement（段階的な関与）
   - 最初はコード不要 → 慣れたらスクリプト編集可能
   - 「スプレッドシートモデル」: 使いながらスキルが向上

3. Composability（合成可能性）
   - モノリシックなアプリではなく、組み合わせ可能なツール
   - 共有データ層による相互運用性

4. Disposability（使い捨て可能性）
   - 生成されたUIは「一度使って終わり」でも構わない
   - 完璧さよりも迅速な反復を重視

5. Transparency（透明性）
   - LLMが生成したコードをユーザーが確認可能
   - 「ブラックボックス」ではなく「ガラスの箱」
```

**注意点:**

> これらの設計原則は研究段階のものが多く、プロダクションレベルで実装されたシステムはまだ少ない状況です。特にComposability（合成可能性）は、現行のSaaS中心のエコシステムとの互換性に課題があります。

## 実現可能性と今後の展望を検討する

### 短期的に実現可能なもの（2026-2027年）

現在のツールチェーンで既に実現可能なMalleable Softwareの形態があります。

**パーソナルスクリプト**: Claude Code、Cursor、GitHub Copilotを使ったワンオフスクリプトの生成は日常的に行われています。データ分析、ファイル変換、API呼び出しの自動化など、特定の個人のニーズに合わせたツールです。

**チーム内ツールのカスタマイズ**: Zapierは社内で800以上のAIエージェントを展開し、89%のAI導入率を達成しています（Anthropic 2026年レポート）。非エンジニアが業務フローを自動化する動きは既に進んでいます。

**プロトタイプの迅速な作成**: Replit、Bolt、Lovableなどのツールにより、非開発者でもWebアプリケーションのプロトタイプを作成できます。

### 中長期的な課題（2027年以降）

**ビジネスモデルの問題**: 合成可能なツールのマネタイズ方法が確立されていません。現在のSaaSモデルは「ロックイン」を前提としており、Malleable Softwareとは根本的に相容れない可能性があります。

**品質保証の自動化**: LLMが生成したコードに対する自動的なセキュリティ検査・テスト生成が、実用レベルに達する必要があります。Rakutenが1,250万行のコードベースを7時間の自律的作業で99.9%の精度で修正した事例は、可能性を示しています（Anthropic 2026年レポート）。

**文化的なシフト**: ユーザーが「ソフトウェアは修正できるもの」と認識するには、教育とツールの両面でのアプローチが必要です。Ink & Switchは「閉じたプラットフォームで形成された環境で、個人のエージェンシーへの欲求を育む」ことを課題として挙げています。

## よくある問題と解決方法

| 問題 | 原因 | 解決方法 |
|------|------|----------|
| AI生成コードにセキュリティ脆弱性がある | LLMが安全でないパターンを学習 | LLMの出力にSAST/DASTツールを組み合わせる |
| 修正したコードが動作しなくなった | コードベース全体の理解不足 | サンドボックス環境でテスト後に適用 |
| 3ヶ月後にコードの意図が不明 | ドキュメントの欠如 | LLMに変更理由のコメント自動生成を依頼 |
| 個人用ツールが組織で共有できない | データ形式の非互換 | MCP等の標準プロトコルを採用 |
| LLMの応答品質が不安定 | プロンプトの曖昧さ | Spec-driven development（仕様駆動開発）を導入 |

## まとめと次のステップ

**まとめ:**

- Malleable Software（可鍛性ソフトウェア）は、LLM・コーディングエージェントの進化により「ユーザーがソフトウェアをコードレベルで修正できる世界」を目指す概念です
- Ink & Switchの3つの設計パターン（Gentle Slope / Tools over Applications / Communal Creation）が理論的な基盤を提供しています
- 2026年現在、Vibe Codingやパーソナルソフトウェアとして部分的に実現が始まっていますが、セキュリティ（AI生成コードの45%が脆弱）、保守性（ロジックエラー1.7倍）、生産性パラドックス（経験者が19%遅くなるケース）といった課題が存在します
- 実現には、LLM基盤に加え、標準化プロトコル（MCP/A2A）、ローカルファーストデータ、サンドボックス実行環境、そして「ソフトウェアは修正できるもの」という文化的シフトが必要です
