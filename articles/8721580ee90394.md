## EntireIOとは：元GitHub CEOが掲げる「コードレビュー時代の終焉」

### 背景：AIエージェント時代の新たな課題

従来のGitワークフローは「人間がコードを書く」前提で設計されました。しかし、Claude CodeやGemini CLIなどのAIエージェントが開発の主役になりつつある2026年現在、以下の新たな課題が顕在化しています:

- **コンテキスト喪失**: エージェントが生成したコードの「なぜ」（プロンプト・思考過程）がコミット履歴に残らない
- **レビュー困難**: コード変更の意図が不明で、人間によるコードレビューが形骸化
- **巻き戻し不能**: エージェントが誤った方向に進んだ際、適切なチェックポイントに戻る手段がない
- **監査・コンプライアンス**: 企業環境で求められる「誰が・なぜ・どのように」の記録が不足

### EntireIOのアプローチ

元GitHub CEOのThomas Dohmke氏は、「**コードを理解しレビューする時代は終わる**」と宣言し、EntireIOを開発しました。EntireIOは以下の特徴を持ちます:

- **Gitワークフロー統合**: `entire enable`コマンドでGitフックを自動設定、既存のGitリポジトリにシームレスに統合
- **AIエージェントセッション記録**: プロンプト・レスポンス・変更ファイル・タイムスタンプを完全記録
- **分離ストレージ**: `entire/checkpoints/v1`ブランチで管理し、通常のコミット履歴を汚さない
- **瞬時巻き戻し**: `entire rewind`で任意のチェックポイントに復元可能
- **並列セッション対応**: Git worktreeと組み合わせ、複数のAIセッションを並行追跡

## 2つの戦略：manual-commit vs auto-commit

EntireIOは、開発スタイルに応じて2つの戦略を提供します。

### manual-commit戦略（デフォルト）

**チェックポイント作成タイミング**: Gitコミット時

```bash
entire enable --strategy manual-commit
```

**特徴**:
- 通常のコミット履歴をクリーンに保つ
- mainブランチでも安全に使用可能
- 巻き戻しは常に非破壊的

**ユースケース**:
- チーム開発で共有するmainブランチ
- コミット履歴を厳密に管理したいプロジェクト
- 手動でコミットタイミングを制御したい場合

**ワークフロー例**:

```bash
# AIエージェントで開発
claude "新しい認証機能を実装して"

# 変更をコミット（このタイミングでチェックポイント作成）
git add .
git commit -m "feat: implement new auth feature"

# セッションが記録される
entire status
# Checkpoints: 1
```

### auto-commit戦略

**チェックポイント作成タイミング**: AIエージェントの各応答後

```bash
entire enable --strategy auto-commit
```

**特徴**:
- エージェント応答ごとに自動コミット作成
- フィーチャーブランチでの使用を推奨（mainブランチでは注意）
- より細かいチェックポイント粒度

**ユースケース**:
- 実験的なフィーチャーブランチ
- AIエージェントの各ステップを詳細追跡したい場合
- 自動コミットが許容されるプロジェクト

**注意点**:

> **auto-commit戦略はmainブランチで使用する際は慎重に**。自動コミットがmainブランチに直接作成されるため、チーム開発では混乱を招く可能性があります。フィーチャーブランチ（`feature/new-auth`等）での使用を推奨します。

### 戦略の比較表

| 観点 | manual-commit | auto-commit |
|------|---------------|-------------|
| コミット作成 | 手動のみ | エージェント応答ごとに自動 |
| mainブランチ安全性 | ✅ 安全 | ⚠️ 注意が必要 |
| チェックポイント粒度 | 粗い（コミット単位） | 細かい（応答単位） |
| rewind動作 | 常に非破壊的 | フィーチャーブランチで完全復元、mainではログのみ |
| 推奨環境 | ほとんどのワークフロー | 実験的フィーチャーブランチ |

## 実践例：rewind・resume・explainコマンド活用

### シナリオ1: エージェントが誤った方向に進んだ場合

Claude Codeに「新しい認証機能を実装して」と指示したところ、意図しないライブラリを使用してしまいました。

```bash
# 現在の状況確認
entire status
# Checkpoints: 3 (最初のコミット、2回目のコミット、3回目のコミット)

# 利用可能なチェックポイント一覧表示
entire rewind
```

**出力例**:

```
Available checkpoints in session 2026-02-16-abc123de...

1. [a3b2c4d5e6f7] 2026-02-16 14:30:00 - feat: add initial auth scaffolding
2. [b4c5d6e7f8a9] 2026-02-16 14:45:00 - feat: implement JWT token validation
3. [c5d6e7f8a9b0] 2026-02-16 15:00:00 - feat: integrate PassportJS (← 誤ったライブラリ)

Select checkpoint to rewind to (1-3):
```

チェックポイント2を選択すると、コードが2の時点に瞬時に復元され、Claude Codeに再度正しい指示を出せます。

### シナリオ2: ブランチ切り替え後にセッション復元

別のフィーチャーブランチ（`feature/payment`）で作業していた最新セッションを復元します。

```bash
# ブランチを切り替えてセッション復元

```

**出力例**:

```
✓ Checked out branch: feature/payment
✓ Restored latest session: 2026-02-15-xyz789ab...
✓ Checkpoints: 5

To continue this session with Claude Code:
  claude resume <session-file>
```

これで、以前の作業コンテキストをそのまま引き継いで開発を再開できます。

### シナリオ3: セッション内容を自然言語で確認

特定のコミットで「なぜその変更を行ったか」を確認したい場合:


**出力例**（AI要約機能が有効な場合）:

## 本番運用のベストプラクティス

### 1. Git worktreeとの併用で並列開発を最適化

EntireIOはGit worktreeと完璧に統合します。複数のフィーチャーブランチで並行してAIセッションを実行できます。

```bash
# メインworktree
cd ~/projects/myapp

# feature/authブランチ用のworktreeを作成
git worktree add ../myapp-auth feature/auth
cd ../myapp-auth
entire enable

# Claude Codeでauth機能を開発
claude "認証機能を実装して"

# 元のディレクトリに戻り、別のworktreeでpayment機能を開発
cd ~/projects/myapp
git worktree add ../myapp-payment feature/payment
cd ../myapp-payment
entire enable

# Gemini CLIでpayment機能を開発
gemini "決済処理を実装して"
```

各worktreeは独立したセッション追跡を持つため、**並行開発による競合は発生しません**。

### 2. summarize機能で監査・コンプライアンス対応

企業環境では「誰が・なぜ・どのように」の記録が求められます。summarize機能を有効化すると、コミット時にAI要約が自動生成されます。

```bash
# .entire/settings.jsonに追記
echo '{
  "strategy_options": {
    "summarize": {
      "enabled": true
    }
  }
}' > .entire/settings.json
```

**必須要件**:
- Claude CLIがインストール・認証済み（`claude`コマンドがPATHにある）

コミット時、以下の要約が自動生成されます:
- **Intent**: 変更の意図
- **Outcome**: 実装結果
- **Learnings**: 学んだ教訓
- **Friction points**: ハマったポイント
- **Open items**: 未解決のタスク

### 3. プロジェクト設定 vs ローカル設定の使い分け

EntireIOは2つの設定ファイルで柔軟な管理が可能です。

**`.entire/settings.json`（プロジェクト設定、Gitコミット対象）**:

```json
{
  "strategy": "manual-commit",
  "agent": "claude-code",
  "enabled": true
}
```

**`.entire/settings.local.json`（ローカル設定、`.gitignore`に追加推奨）**:

```json
{
  "enabled": false,
  "log_level": "debug"
}
```

**使い分けの例**:
- チーム全体で`manual-commit`戦略を採用 → `settings.json`に記載
- 個人的にデバッグログを有効化したい → `settings.local.json`に記載

ローカル設定は**フィールド単位で**プロジェクト設定を上書きします。

### 4. テレメトリーとプライバシー

EntireIOはデフォルトで匿名使用統計をPosthogに送信します（コード内容は送信されません）。無効化する場合:

```bash
entire enable --telemetry=false
```

または設定ファイルで:

```json
{
  "telemetry": false
}
```

## よくある問題と解決方法

| 問題 | 原因 | 解決方法 |
|------|------|----------|
| `Not a git repository` | Gitリポジトリ外で実行 | `git init`でリポジトリ初期化、または既存リポジトリに移動 |
| `Entire is disabled` | EntireIOが有効化されていない | `entire enable`を実行 |
| `No rewind points found` | チェックポイントが作成されていない | Claude Codeで作業後、`git commit`（manual-commit）またはエージェント応答を待つ（auto-commit） |
| `shadow branch conflict` | `entire/checkpoints/v1`ブランチが競合 | `entire reset --force`で強制リセット |
| SSH認証エラー（`git push`時） | go-gitのSSH処理の既知の問題 | GitHubホストキーを`known_hosts`に追加（詳細はREADME参照） |
