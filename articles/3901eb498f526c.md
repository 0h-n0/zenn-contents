# LangGraph Store APIで実装するマルチエージェントRAGの共有メモリと長期記憶

## この記事でわかること

- LangGraphのStore APIを使った**マルチエージェント間メモリ共有**の名前空間設計パターン
- Checkpointer（短期メモリ）とStore（長期メモリ）の**2層アーキテクチャ**をRAGに適用する方法
- LangMem SDKによる**セマンティック・エピソード・手続き記憶**の自動抽出と管理
- PostgresStore + pgvectorを使った**ベクトル検索付き長期記憶**の本番実装
- Claude Sonnet 4.6のcontext compactionと組み合わせた**メモリ効率の最適化**手法

- **想定読者**: 中級〜上級のPythonエンジニアでLLMエージェント開発経験者
- **必要な前提知識**:
  - Python 3.11+の非同期プログラミング（async/await）
  - LangGraph 1.0の基本概念（StateGraph、ノード、エッジ）
  - RAG（Retrieval-Augmented Generation）の基本アーキテクチャ
  - Claude APIの基本的な利用経験

## 結論・成果

LangGraph 1.0のStore APIとLangMem SDKを組み合わせることで、マルチエージェントRAGに**クロスエージェント長期記憶**を実装できます。公式ドキュメントによると、Store APIの名前空間ベース設計により、エージェント間のメモリ分離と共有を柔軟に制御可能です。

本記事で構築するアーキテクチャの特徴は以下の通りです。

| 指標 | 従来（State共有のみ） | Store API + LangMem導入後 |
|------|---------------------|--------------------------|
| メモリスコープ | スレッド内のみ | クロススレッド・クロスエージェント |
| メモリ永続性 | グラフ実行中のみ | PostgreSQL/MongoDBで永続化 |
| 検索方式 | キー完全一致 | ベクトル類似度検索 + メタデータフィルタ |
| メモリタイプ | 会話履歴のみ | セマンティック・エピソード・手続きの3種 |
| 自動抽出 | なし | LangMemバックグラウンドマネージャー |

> **制約**: Store APIはLangGraph 1.0以降で利用可能です。0.x系からのマイグレーションでは `langgraph.prebuilt` が非推奨となり `langchain.agents` に移行されている点に注意してください。また、ベクトル検索にはpgvector拡張（PostgresStore）またはAtlas Vector Search（MongoDBStore）が必要です。

## マルチエージェントRAGのメモリアーキテクチャを設計する

マルチエージェントRAGでは、複数の専門エージェント（Query Analyzer、Retriever、Verifier、Synthesizer等）が協調して回答を生成します。このとき、**エージェント間でどの情報を共有し、どの情報を分離するか**がシステムの品質を左右します。

### 2層メモリの役割分担を理解する

LangGraphは**短期メモリ（Checkpointer）**と**長期メモリ（Store）**を明確に分離した2層アーキテクチャを採用しています。公式ドキュメントによると、「Checkpointerはグラフのスーパーステップごとにステートのスナップショットを保存する」仕組みです。

```python
# memory_architecture.py
from langgraph.checkpoint.postgres import PostgresSaver
from langgraph.store.postgres import PostgresStore
from langchain_anthropic import ChatAnthropic

# 短期メモリ: スレッドスコープの会話状態
# 1つのRAGセッション内のクエリ履歴・検索結果を保持
checkpointer = PostgresSaver.from_conn_string(
    "postgresql://user:pass@localhost:5432/rag_memory"
)

# 長期メモリ: クロススレッド・クロスエージェントの永続記憶
# 過去のセッションで学んだ知識・ユーザー嗜好・成功パターンを保持
store = PostgresStore.from_conn_string(
    "postgresql://user:pass@localhost:5432/rag_memory",
    index={
        "dims": 1024,
        "embed": lambda texts: anthropic_embed(texts),  # 後述の実装
        "fields": ["content", "metadata.summary"],
    },
)

llm = ChatAnthropic(
    model="claude-sonnet-4-6-20250514",
    max_tokens=8192,
)
```

**なぜ2層に分離するのか:**
- **短期メモリ（Checkpointer）**: 1回のRAGセッションの中間状態（クエリ解析結果、検索ドキュメント）を保持。セッション終了後は不要
- **長期メモリ（Store）**: セッションを跨いで有用な知識（検索傾向、クエリ書き換えパターン）を蓄積

> 最初は短期メモリだけで十分と考えましたが、RAGセッションを重ねるうちに「同じクエリパターンで同じ失敗を繰り返す」問題が発生します。長期メモリに成功パターンを蓄積することで回避できます。

### 名前空間設計でメモリの分離と共有を制御する

Store APIの名前空間はタプルベースの階層構造を採用しています。マルチエージェントRAGでは、この名前空間設計が**メモリの可視性制御**の鍵になります。

```python
# namespace_design.py
from langgraph.store.base import BaseStore

def design_rag_namespaces(user_id: str, session_id: str) -> dict[str, tuple[str, ...]]:
    """マルチエージェントRAGの名前空間設計

    3層の名前空間でメモリの可視性を制御:
    - グローバル: 全エージェントが読み書き可能
    - エージェント固有: 特定エージェントのみ読み書き可能
    - ユーザー固有: ユーザーごとに分離された個人設定
    """
    return {
        # 全エージェント共有: 検索結果のキャッシュ、ドキュメントメタデータ
        "shared_knowledge": ("rag", "shared", "knowledge"),
        # 全エージェント共有: クエリ書き換え成功パターン
        "shared_patterns": ("rag", "shared", "query_patterns"),
        # Query Analyzer専用: クエリ分類ルール
        "query_analyzer": ("rag", "agents", "query_analyzer"),
        # Retriever専用: インデックス選択履歴
        "retriever": ("rag", "agents", "retriever"),
        # Verifier専用: 検証基準・閾値
        "verifier": ("rag", "agents", "verifier"),
        # Synthesizer専用: 回答テンプレート・トーン設定
        "synthesizer": ("rag", "agents", "synthesizer"),
        # ユーザー固有: 検索嗜好・フィードバック履歴
        "user_preferences": ("rag", "users", user_id, "preferences"),
        # セッション固有: 現在のセッションのメモリ（TTLで自動削除）
        "session_cache": ("rag", "sessions", session_id),
    }
```

**なぜ3層の名前空間が必要か:**

| 層 | スコープ | 用途 | TTL |
|----|---------|------|-----|
| グローバル共有 | 全エージェント | 検索キャッシュ、成功パターン | なし（永続） |
| エージェント固有 | 各エージェント | 専門知識、チューニング済みパラメータ | なし（永続） |
| ユーザー固有 | ユーザーごと | 検索嗜好、フィードバック | 90日 |
| セッション固有 | セッションごと | 中間計算結果のキャッシュ | 24時間 |

**注意点:**
> 名前空間のタプルには空文字やドットを含めることはできません。また、`("langgraph",)` で始まるプレフィックスはシステム予約されているため使用できません。公式ドキュメントに明記されている制約です。

## Store APIでクロスエージェントメモリ共有を実装する

実際にStore APIを使って、マルチエージェントRAGの各エージェントがメモリを読み書きする実装を見ていきましょう。

### エージェント共有メモリの読み書きパターン

```python
# shared_memory.py
from typing import Annotated, TypedDict
from langgraph.store.base import BaseStore
from langgraph.graph import StateGraph
from langchain_core.runnables import RunnableConfig
import uuid

class RAGState(TypedDict):
    """マルチエージェントRAGの共有State"""
    query: str
    rewritten_query: str
    search_results: list[dict]
    verified_results: list[dict]
    answer: str
    quality_score: float

async def query_analyzer_node(
    state: RAGState,
    config: RunnableConfig,
    *,
    store: BaseStore,
) -> dict:
    """Query Analyzerノード: クエリを解析し、過去の成功パターンを活用"""
    query = state["query"]

    # 共有メモリから過去のクエリ書き換えパターンを検索（セマンティック検索）
    past_patterns = await store.asearch(
        ("rag", "shared", "query_patterns"),
        query=query,
        limit=5,
    )

    # エージェント固有メモリからクエリ分類ルールを取得
    classification_rules = await store.aget(
        ("rag", "agents", "query_analyzer"),
        "classification_rules",
    )

    # Claude Sonnet 4.6でクエリを解析・書き換え
    rewritten = await analyze_and_rewrite(
        query=query,
        past_patterns=[p.value for p in past_patterns],
        rules=classification_rules.value if classification_rules else None,
        llm=llm,
    )

    return {"rewritten_query": rewritten}


async def retriever_node(
    state: RAGState,
    config: RunnableConfig,
    *,
    store: BaseStore,
) -> dict:
    """Retrieverノード: 検索を実行し、結果をメモリに保存"""
    rewritten_query = state["rewritten_query"]

    # 共有メモリからドキュメントメタデータキャッシュを検索
    cached_docs = await store.asearch(
        ("rag", "shared", "knowledge"),
        query=rewritten_query,
        limit=3,
    )

    # キャッシュヒット率が高い場合はキャッシュを活用
    if cached_docs and cached_docs[0].score > 0.9:
        results = [doc.value for doc in cached_docs]
    else:
        # ベクトルストアから検索を実行
        results = await vector_store_search(rewritten_query)

        # 検索結果を共有メモリに保存（次回以降のキャッシュ）
        for doc in results:
            await store.aput(
                ("rag", "shared", "knowledge"),
                str(uuid.uuid4()),
                {
                    "content": doc["content"][:500],  # 先頭500文字をインデックス
                    "metadata": {
                        "source": doc["source"],
                        "summary": doc.get("summary", ""),
                        "retrieved_at": "2026-02-23",
                    },
                },
            )

    return {"search_results": results}


async def synthesizer_node(
    state: RAGState,
    config: RunnableConfig,
    *,
    store: BaseStore,
) -> dict:
    """Synthesizerノード: 検索結果から回答を生成し、成功パターンを記録"""
    verified_results = state["verified_results"]
    original_query = state["query"]
    rewritten_query = state["rewritten_query"]

    # ユーザー固有メモリから回答スタイル設定を取得
    user_id = config.get("configurable", {}).get("user_id", "default")
    user_prefs = await store.aget(
        ("rag", "users", user_id, "preferences"),
        "answer_style",
    )

    answer = await generate_answer(
        query=original_query,
        results=verified_results,
        style=user_prefs.value if user_prefs else None,
        llm=llm,
    )

    # 成功パターンを共有メモリに保存（他のエージェントも活用可能）
    if state.get("quality_score", 0) > 0.8:
        await store.aput(
            ("rag", "shared", "query_patterns"),
            str(uuid.uuid4()),
            {
                "content": f"Original: {original_query} -> Rewritten: {rewritten_query}",
                "metadata": {
                    "quality_score": state["quality_score"],
                    "result_count": len(verified_results),
                },
            },
        )

    return {"answer": answer}
```

**なぜこの実装を選んだか:**
- `store.asearch()` でセマンティック検索を使うことで、**完全一致ではなく意味的に類似した過去パターン**を発見できます
- 名前空間の分離により、各エージェントは自分の専門メモリにのみ書き込み、共有メモリからは読み取りを行う**Read-Heavy / Write-Selective**パターンを実現します
- `uuid.uuid4()` をキーに使うことで、同一名前空間に複数のメモリを蓄積できます

### グラフのコンパイルとStore注入

```python
# graph_compile.py
from langgraph.graph import StateGraph, END

def build_rag_graph() -> StateGraph:
    """マルチエージェントRAGグラフを構築"""
    graph = StateGraph(RAGState)

    graph.add_node("query_analyzer", query_analyzer_node)
    graph.add_node("retriever", retriever_node)
    graph.add_node("verifier", verifier_node)  # 実装省略
    graph.add_node("synthesizer", synthesizer_node)

    graph.set_entry_point("query_analyzer")
    graph.add_edge("query_analyzer", "retriever")
    graph.add_edge("retriever", "verifier")
    graph.add_edge("verifier", "synthesizer")
    graph.add_edge("synthesizer", END)

    return graph

# Checkpointer + Store の両方を注入してコンパイル
app = build_rag_graph().compile(
    checkpointer=checkpointer,  # 短期メモリ（セッション状態）
    store=store,                # 長期メモリ（クロスエージェント共有）
)

# 実行時にthread_idとuser_idを指定
result = await app.ainvoke(
    {"query": "LangGraphのStore APIでメモリ共有するには？"},
    config={
        "configurable": {
            "thread_id": "session-001",
            "user_id": "user-abc",
        }
    },
)
```

**注意点:**
> `compile()` 時に `store` を渡すと、すべてのノード関数で `store: BaseStore` 引数が利用可能になります。この依存注入パターンにより、テスト時には `InMemoryStore` に差し替えることで外部データベースなしでテストできます。

## LangMem SDKでメモリの自動抽出と管理を実装する

Store APIは低レベルの読み書きインターフェースですが、**何をメモリとして保存すべきか**の判断はアプリケーション側で実装する必要があります。LangMem SDK（v0.0.30）はこの判断を自動化するツールキットです。

### 3種類のメモリタイプを使い分ける

LangMem SDKは人間の記憶モデルに基づき、3種類のメモリタイプを提供しています。

| メモリタイプ | 内容 | RAGでの用途 | 保存先名前空間例 |
|------------|------|-----------|----------------|
| **セマンティック記憶** | 事実・概念・知識 | ドキュメントメタデータ、エンティティ情報 | `("rag", "semantic", entity_type)` |
| **エピソード記憶** | 過去の経験・イベント | 成功した検索パターン、ユーザー質問履歴 | `("rag", "episodic", user_id)` |
| **手続き記憶** | 操作手順・ルール | システムプロンプトの改善、エージェント動作ルール | `("rag", "procedural", agent_id)` |

```python
# langmem_integration.py
from langmem import create_manage_memory_tool, create_search_memory_tool
from langgraph.store.memory import InMemoryStore
from langgraph.prebuilt import create_react_agent

# 開発環境: InMemoryStore（本番ではPostgresStoreに差し替え）
memory_store = InMemoryStore(
    index={
        "dims": 1024,
        "embed": embed_fn,  # Voyage AI or OpenAI embedding
        "fields": ["content"],
    }
)

# メモリ管理ツール: エージェントが会話中にメモリを保存
manage_memory = create_manage_memory_tool(
    namespace=("rag", "semantic", "entities"),
)

# メモリ検索ツール: エージェントが過去のメモリを検索
search_memory = create_search_memory_tool(
    namespace=("rag", "semantic", "entities"),
)

# RAGエージェントにメモリツールを追加
rag_agent = create_react_agent(
    model=llm,
    tools=[retrieve_tool, manage_memory, search_memory],
    store=memory_store,
    checkpointer=checkpointer,
)
```

### バックグラウンドメモリマネージャーで非同期抽出を実装する

RAGの応答レイテンシを悪化させずにメモリを蓄積するには、**バックグラウンド処理**が有効です。LangMem SDKのバックグラウンドマネージャーは、会話終了後に非同期でメモリを抽出・統合します。

```python
# background_memory.py
from langmem import create_memory_store_manager

# バックグラウンドメモリマネージャーの設定
memory_manager = create_memory_store_manager(
    store=store,
    model=llm,
    # セマンティック記憶: 会話から事実・エンティティを自動抽出
    semantic_namespace=("rag", "semantic", "auto_extracted"),
    # エピソード記憶: 検索セッションの要約を保存
    episodic_namespace=("rag", "episodic", "sessions"),
    # 手続き記憶: エージェント動作の改善ルールを蓄積
    procedural_namespace=("rag", "procedural", "improvements"),
)

async def on_session_complete(
    session_messages: list[dict],
    session_metadata: dict,
) -> None:
    """RAGセッション完了時にバックグラウンドでメモリ抽出"""
    await memory_manager.process(
        messages=session_messages,
        metadata={
            "session_id": session_metadata["session_id"],
            "user_id": session_metadata["user_id"],
            "quality_score": session_metadata.get("quality_score"),
        },
    )
```

**なぜバックグラウンド処理を選んだか:**
- Hot Pathでメモリを書き込むとLLMへの追加呼び出しで**レイテンシが200-500ms増加**します
- バックグラウンドでは応答完了後に非同期処理するため**レイテンシに影響しません**
- ただし、保存メモリが次のリクエストで即座に利用できない場合があります。リアルタイム性が必要ならHot Pathとの併用を検討してください

## PostgresStore + pgvectorで本番環境の長期記憶を構築する

開発環境の`InMemoryStore`はプロセス終了でデータが消失するため、本番環境では**PostgresStore**を使用します。pgvector拡張によるベクトル検索が組み込まれており、セマンティックメモリの検索に適しています。

### 本番用PostgresStoreのセットアップ

```python
# production_store.py
from langgraph.store.postgres import AsyncPostgresStore
import psycopg

# 非同期版PostgresStoreの初期化（本番推奨）
async def create_production_store() -> AsyncPostgresStore:
    """本番環境用のPostgresStoreを作成

    PostgreSQL 15+ と pgvector 拡張が必要:
    CREATE EXTENSION IF NOT EXISTS vector;
    """
    store = AsyncPostgresStore.from_conn_string(
        "postgresql://rag_user:secure_pass@db.example.com:5432/rag_memory",
        index={
            # Voyage AI embedding (1024次元)
            "dims": 1024,
            "embed": voyage_embed_fn,
            # インデックス対象フィールド
            "fields": ["content", "metadata.summary"],
        },
    )
    # テーブル・インデックスの自動作成
    await store.setup()
    return store
```

### ベクトルインデックスの選択基準

PostgresStoreはpgvectorの3種類のANNインデックスをサポートしています。公式ドキュメントによると、用途に応じた選択基準は以下の通りです。

| インデックス | 構築速度 | 検索精度（Recall） | メモリ使用量 | 推奨用途 |
|------------|---------|------------------|------------|---------|
| **HNSW** | 遅い | 高い（99%+） | 多い | **大半のRAGアプリ**（推奨） |
| **IVFFlat** | 速い | 中程度（95%程度） | 少ない | メモリ数100万件以上 |
| **Flat** | なし | 100%（完全一致） | 最小 | メモリ数1万件以下 |

**注意点:**
> HNSWインデックスの `m` パラメータ（デフォルト16）と `ef_construction`（デフォルト64）は、メモリ数が10万件を超える場合にチューニングが必要です。`m` を増やすと検索精度は向上しますが、インデックスサイズが増大します。本番投入前にベンチマークを取ることを推奨します。

### TTLによるメモリの自動クリーンアップ

長期記憶は無制限に蓄積するとストレージコストと検索精度の両方に悪影響を与えます。PostgresStoreのTTL機能で、用途に応じた自動クリーンアップを設定します。

```python
# ttl_management.py
# セッションキャッシュ: 24時間で自動削除
await store.aput(
    ("rag", "sessions", "session-001"),
    "intermediate_results",
    {"rewritten_query": "...", "top_docs": [...]},
    ttl=1440,
)

# ユーザー嗜好: 90日で自動削除
await store.aput(
    ("rag", "users", "user-abc", "preferences"),
    "answer_style",
    {"tone": "technical", "length": "detailed"},
    ttl=129600,
)

# 共有知識: 永続保存（ttl省略）
await store.aput(
    ("rag", "shared", "knowledge"),
    "doc-metadata-001",
    {"content": "LangGraph Store APIの概要...", "metadata": {"source": "official_docs"}},
)
```

## Claude Sonnet 4.6のcontext compactionとメモリ効率を最適化する

マルチエージェントRAGが長時間稼働すると、蓄積されたメモリがコンテキストウィンドウを圧迫します。Claude Sonnet 4.6の**context compaction**（ベータ）は、古いコンテキストを自動要約して有効コンテキスト長を拡大する機能です。

### メモリ階層とcontext compactionの連携

```python
# context_optimization.py
from langchain_anthropic import ChatAnthropic

# Claude Sonnet 4.6: 200K標準 / 1Mベータ
llm = ChatAnthropic(
    model="claude-sonnet-4-6-20250514",
    max_tokens=8192,
    # context compactionを有効化（ベータ）
    model_kwargs={
        "metadata": {"context_compaction": True},
    },
)

async def memory_aware_generation(
    store: AsyncPostgresStore,
    query: str,
    user_id: str,
) -> str:
    """メモリ階層を活用した回答生成

    メモリの優先順位:
    1. セッション内メモリ（Checkpointer経由、最高優先度）
    2. ユーザー固有メモリ（Store検索、中優先度）
    3. グローバル共有メモリ（Store検索、低優先度）
    """
    # ユーザー固有のエピソード記憶を検索
    user_episodes = await store.asearch(
        ("rag", "episodic", user_id),
        query=query,
        limit=3,
    )

    # グローバル共有のセマンティック記憶を検索
    shared_knowledge = await store.asearch(
        ("rag", "semantic", "auto_extracted"),
        query=query,
        limit=5,
    )

    # メモリをコンテキストに組み込み（優先度順）
    memory_context = format_memories(
        user_episodes=[e.value for e in user_episodes],
        shared_knowledge=[k.value for k in shared_knowledge],
    )

    # Claude Sonnet 4.6のcontext compactionが古いコンテキストを自動圧縮
    response = await llm.ainvoke(
        [
            {"role": "system", "content": build_system_prompt(memory_context)},
            {"role": "user", "content": query},
        ]
    )

    return response.content


def format_memories(
    user_episodes: list[dict],
    shared_knowledge: list[dict],
) -> str:
    """メモリを構造化テキストに変換"""
    sections = []
    if user_episodes:
        episodes = "\n".join(f"- {ep.get('content', '')}" for ep in user_episodes)
        sections.append(f"## 過去の関連セッション\n{episodes}")
    if shared_knowledge:
        knowledge = "\n".join(f"- {kn.get('content', '')}" for kn in shared_knowledge)
        sections.append(f"## 関連知識\n{knowledge}")
    return "\n\n".join(sections)
```

**注意点:**
> context compactionはベータ機能であり、本番環境での利用は慎重に検討してください。公式ドキュメントによると、「古いコンテキストを自動要約して有効コンテキスト長を拡大する」機能ですが、要約時に情報が失われる可能性があります。重要なメモリはStore APIに永続化し、コンテキストウィンドウには最新・最関連の情報のみを渡す設計が安全です。

## よくある問題と解決方法

マルチエージェントRAGのメモリ共有で頻出する問題とその対処法を整理します。

| 問題 | 原因 | 解決方法 |
|------|------|----------|
| メモリ検索が遅い（>100ms） | pgvectorインデックス未作成 | `CREATE INDEX` でHNSWインデックスを作成 |
| 異なるエージェントが同じメモリを重複保存 | 名前空間設計の不備 | エージェントごとの書き込み名前空間を分離 |

| `InMemoryStore` でデータが消える | プロセス再起動 | 本番は `PostgresStore` / `MongoDBStore` を使用 |
| Store注入後にノードでstoreが取得できない | `compile(store=...)` 未指定 | グラフの `compile()` 時にstoreを渡す |

### メモリの競合を防ぐ設計指針

マルチエージェントが並列実行される場合、同一名前空間への同時書き込みで競合が発生する可能性があります。

```python
# conflict_prevention.py
import uuid
from datetime import datetime, timezone

async def safe_memory_write(
    store: AsyncPostgresStore,
    namespace: tuple[str, ...],
    value: dict,
    agent_id: str,
) -> None:
    """Append-onlyパターンで競合を防ぐメモリ書き込み"""
    key = f"{agent_id}_{uuid.uuid4().hex[:12]}"
    value_with_metadata = {
        **value,
        "_metadata": {
            "agent_id": agent_id,
            "created_at": datetime.now(timezone.utc).isoformat(),
        },
    }
    await store.aput(namespace, key, value_with_metadata)
```

**なぜAppend-onlyパターンを採用するか:**
- `put()` は同一キーで上書きするため、複数エージェントが同じキーに書き込むと**最後の書き込みのみが残ります**
- UUIDベースのキーにより各エージェントのメモリが独立して蓄積されます
- Store APIは現時点でCAS操作をサポートしていないため、Append-onlyが最もシンプルです
