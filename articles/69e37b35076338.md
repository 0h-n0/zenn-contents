## Guidanceとは何か：従来アプローチの課題

### LLM出力の制御が困難な理由

LLM（大規模言語モデル）に「JSONで出力してください」とプロンプトで指示しても、実際には以下のような問題が発生します：

- 余計なテキストが混入（例: "Here is the JSON:"という前置き）
- JSONの構文エラー（閉じ括弧の欠落、ダブルクォートの不整合）
- スキーマ違反（指定していないフィールドの追加、型の不一致）

実際のプロダクション環境では、**LLM出力の約40%が何らかの形式エラーを含む**というデータもあります。

## Guidanceの仕組み：トークン単位での制御

### アーキテクチャの核心

Guidanceは**推論層でトークン単位でLLMを制御**することで、構造化出力を保証します。

```
従来: プロンプト → LLM → 出力 → 検証 → ❌失敗 → 再試行
Guidance: プロンプト → LLM（トークン制約付き） → ✅正確な出力
```

具体的には、以下の3つの制約方法を提供します：

| 制約方法 | 用途 | 例 |
|---------|------|-----|
| **select** | 選択肢の制限 | `select(["A", "B", "C"])` |
| **正規表現** | パターン指定 | `gen(regex=r"\d{3}-\d{4}")` |
| **文脈自由文法（CFG）** | 複雑な構造 | JSON、Python、SQL等 |

### 実装例1：JSONスキーマ制約

```python
# guidance_json_example.py
from guidance import models, json as gen_json
from pydantic import BaseModel

# Pydanticモデルでスキーマ定義
class UserProfile(BaseModel):
    name: str
    age: int
    email: str
    role: str  # "admin" or "user"

# OpenAI GPT-4モデルをロード
lm = models.OpenAI("gpt-4o")

# Guidanceプログラム（lm += パターンで直接制約）
lm += "Extract user information from this text: John Doe, 30, john@example.com, admin\n"
lm += gen_json(name="user", schema=UserProfile)  # ← Pydanticスキーマで制約

# 結果をPydanticオブジェクトに変換
user = UserProfile.model_validate_json(lm["user"])
print(user.name)  # "John Doe"
print(user.age)   # 30
```

**なぜこの実装を選んだか:**
- Pydanticによる型安全性（IDEの自動補完が効く）
- 再試行なしで100%の成功率
- API呼び出しが1回のみ（コスト削減）

**注意点:**
> Guidanceは制約付き生成のため、通常のプロンプティングより**若干遅い**（約10〜20%のオーバーヘッド）。ただし、再試行コストを考慮すると全体では30〜50%高速化します。

- **柔軟性最優先**: Instructor（カスタム検証ロジック）
- **プログラマブル制御**: LMQL（独自言語）

### Outlinesとの技術的な違い

**Outlines**は「有限状態機械（FSM）の事前コンパイル」により、推論中のルックアップをO(c)定数時間で実現します。これにより、Guidanceより**2〜3倍高速**です。

**ただし、Outlinesの制約**:
- インライン生成（プロンプト中間での生成）に非対応
- 複数の生成結果を辞書キーで管理できない
- 開発UXがGuidanceより劣る

```python
# Outlines: インライン生成が不可能
# ❌ できないこと
"""
ステップ1を生成 → その結果を使ってステップ2を生成
"""

# Guidance: インライン生成が可能
# ✅ できること
with models.Context(lm) as lm:
    lm += "Step 1: "
    lm += gen("step1", max_tokens=50)
    lm += f"Based on {lm['step1']}, Step 2: "
    lm += gen("step2", max_tokens=50)
```

**結論**: 複数ステップの推論が必要ならGuidance、単純な構造化出力ならOutlinesが最適。

## 本番運用で直面する3つの課題と解決策

### 課題1：デバッグの難しさ

**問題**: Guidanceは推論層で制約を適用するため、「なぜそのトークンが選ばれたか」が不透明です。

**解決策**: ログ出力とステップバイステップ実行

```python
# guidance_debug.py
from guidance import models, gen
import logging

logging.basicConfig(level=logging.DEBUG)  # デバッグログ有効化

lm = models.OpenAI("gpt-4o")

with models.Context(lm) as lm:
    lm += "Generate a number: "
    lm += gen(regex=r"\d+", name="number")
    print(f"Generated: {lm['number']}")  # 中間結果を確認
```

### 課題2：制約が複雑すぎると生成失敗

**問題**: 正規表現やCFGが厳しすぎると、LLMが有効なトークンを見つけられず無限ループに陥ることがあります。

**解決策**: タイムアウトと制約の段階的緩和

```python
# guidance_timeout.py
from guidance import models, gen

lm = models.OpenAI("gpt-4o")

try:
    with models.Context(lm, timeout=10) as lm:  # 10秒でタイムアウト
        lm += gen(regex=r"very-strict-pattern")
except TimeoutError:
    # フォールバック: 制約を緩和
    with models.Context(lm) as lm:
        lm += gen(regex=r"relaxed-pattern")
```

### 課題3：コスト vs 速度のトレードオフ

**問題**: Guidanceは単一API呼び出しだが、制約処理のオーバーヘッドで若干遅い。

**解決策**: ハイブリッドアプローチ

```python
# guidance_hybrid.py
from guidance import models, gen

# 簡単な出力: 通常のプロンプト（高速）
simple_response = openai.chat.completions.create(...)

# 複雑な構造化出力: Guidance（高信頼性）
lm = models.OpenAI("gpt-4o")
with models.Context(lm) as lm:
    lm += gen(json=ComplexSchema)
```

## よくある問題と解決方法

| 問題 | 原因 | 解決方法 |
|------|------|----------|
| `JSONDecodeError` | CFG定義のミス | Pydanticスキーマでバリデーション |
