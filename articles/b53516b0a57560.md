## Wan-VAEのアーキテクチャを理解する

Wan2.2の動画生成パイプラインを最適化するには、まずVAEコンポーネントの構造を把握する必要があります。Wan-VAEは127Mパラメータの比較的コンパクトなモデルですが、動画の時空間圧縮において重要な役割を果たしています。

### 3D Causal ConvolutionとU-Net構造

Wan-VAEは**U-Net + 3D Causal Convolution**をベースとしたアーキテクチャを採用しています。通常の3D畳み込みと異なり、Causal Convolution（因果畳み込み）は未来のフレーム情報を参照しないように設計されており、時間方向の厳密な因果性を保証します。

```python
# Wan-VAEの潜在空間への圧縮イメージ
# 入力: (1+T) x H x W x 3 の動画テンソル
# 出力: C x (1+T/4) x (H/8) x (W/8) の潜在表現

# Wan2.1-VAE: 時間4x, 空間8x8 = 4x8x8 圧縮
# Wan2.2-VAE: 時間4x, 空間16x16 = 4x16x16 圧縮（5Bモデル用）

import torch

# 圧縮比の違いを確認する例
input_shape = (1, 3, 81, 720, 1280)  # B, C, T, H, W (81フレーム, 720p)

# Wan2.1-VAE の場合
latent_21 = (1, 16, 21, 90, 160)  # T/4, H/8, W/8
compression_21 = (81*720*1280*3) / (16*21*90*160)
print(f"Wan2.1-VAE 圧縮率: {compression_21:.1f}x")  # 約46.3x

# Wan2.2-VAE の場合（5Bモデル）
latent_22 = (1, 16, 21, 45, 80)  # T/4, H/16, W/16
compression_22 = (81*720*1280*3) / (16*21*45*80)
print(f"Wan2.2-VAE 圧縮率: {compression_22:.1f}x")  # 約185.1x
```

この高い圧縮比により、5BモデルはRTX 4090のような消費者向けGPUでも720P/24fpsの動画生成に対応しています。

**注意点:**
> Wan2.2-VAEの16×16×4圧縮は5B TI2Vモデル専用です。14Bモデル（T2V-A14B、I2V-A14B）はWan2.1-VAE（8×8×4圧縮）を使用します。ファインチューニング対象のモデルに応じてVAEを選択してください。

### RMSNormとGroupNormの置き換え

Wan-VAEでは正規化層に**RMSNorm**を採用し、従来のGroupNormを置き換えています。公式技術文書によると、この変更には2つの目的があります。

1. **時間方向の因果性の保持**: GroupNormはバッチ内の統計量を使うため、時間方向の因果性を崩す可能性があります
2. **メモリ効率的な推論**: チャンク単位で動画を処理する際のメモリ消費を抑制します

### VAE損失関数の構成

Wan-VAEの学習では、3つの損失関数を組み合わせた複合損失を使用します。

$$
\mathcal{L}_{\text{VAE}} = \lambda_{\ell_1} \| V - \hat{V} \|_1 + \lambda_{\text{perc}} \cdot \text{LPIPS}(V, \hat{V}) + \lambda_{\text{KL}} \cdot D_{\text{KL}}(q(z|V) \| p(z))
$$

各項のバランスは以下の通りです。

| 損失項 | 重み | 役割 |
|--------|------|------|
| L1再構成損失 | $\lambda_{\ell_1} = 3$ | ピクセルレベルの再構成精度 |
| LPIPS知覚損失 | $\lambda_{\text{perc}} = 3$ | 人間の知覚に近い品質評価 |
| KLダイバージェンス | $\lambda_{\text{KL}} = 3 \times 10^{-6}$ | 潜在空間の正則化 |

KL項の重みが極めて小さい（$3 \times 10^{-6}$）点が特徴的で、再構成品質を優先しつつ潜在空間の構造を維持する設計です。公式の報告では、720×720px・25fpsの動画に対して**PSNR約33dB**を達成しています。

## VAEファインチューニングの3段階パイプラインを設計する

Wan-VAEの学習は3段階のパイプラインで構成されています。VAE自体のファインチューニングを行う場合も、この段階的アプローチに従う必要があります。

### Stage 1: 2D画像VAEの学習

最初に画像ドメインで2D VAEを学習します。この段階では時間方向の処理を含まず、空間的な圧縮と再構成に集中します。

```python
# Stage 1: 2D VAE学習の概念コード
# 注意: Wan2.2公式のVAE学習コードは未公開のため、
# 一般的なVAE学習パイプラインを参考に構成

import torch
import torch.nn.functional as F
from torchvision import transforms
from torch.utils.data import DataLoader

# 画像データセット用の前処理
transform = transforms.Compose([
    transforms.Resize((256, 256)),
    transforms.ToTensor(),
    transforms.Normalize(mean=[0.5, 0.5, 0.5], std=[0.5, 0.5, 0.5])
])

def compute_vae_loss(
    original: torch.Tensor,
    reconstructed: torch.Tensor,
    mu: torch.Tensor,
    logvar: torch.Tensor,
    lpips_fn,  # LPIPS損失関数
    lambda_l1: float = 3.0,
    lambda_perc: float = 3.0,
    lambda_kl: float = 3e-6,
) -> torch.Tensor:
    """Wan-VAE損失関数の実装"""
    # L1再構成損失
    l1_loss = F.l1_loss(reconstructed, original)

    # LPIPS知覚損失
    perc_loss = lpips_fn(reconstructed, original).mean()

    # KLダイバージェンス
    kl_loss = -0.5 * torch.mean(1 + logvar - mu.pow(2) - logvar.exp())

    total = lambda_l1 * l1_loss + lambda_perc * perc_loss + lambda_kl * kl_loss
    return total
```

### Stage 2: 3D Inflation（2D → 3D拡張）

2D VAEの重みを3D構造に「膨張（inflate）」させます。2D畳み込みカーネルを時間軸方向に複製し、3D Causal Convolutionに変換します。

```python
# Stage 2: 2D → 3D Inflation の概念
# Conv2d (kernel: H x W) → Conv3d (kernel: T x H x W)

def inflate_conv2d_to_conv3d(conv2d_weight: torch.Tensor, temporal_size: int = 3) -> torch.Tensor:
    """2D畳み込み重みを3D Causal Convolutionに拡張する

    Args:
        conv2d_weight: shape (out_ch, in_ch, H, W)
        temporal_size: 時間方向のカーネルサイズ

    Returns:
        conv3d_weight: shape (out_ch, in_ch, T, H, W)
    """
    out_ch, in_ch, h, w = conv2d_weight.shape
    # 時間方向に複製し、合計を保存するため除算
    conv3d_weight = conv2d_weight.unsqueeze(2).repeat(1, 1, temporal_size, 1, 1)
    conv3d_weight = conv3d_weight / temporal_size
    return conv3d_weight

# Inflation後、短い動画（5フレーム、128x128）で微調整
# この段階で時間方向の整合性を学習する
```

この段階では**5フレーム・128×128の短い動画クリップ**で微調整を行い、時間方向の連続性を学習させます。

### Stage 3: GAN損失によるファインチューニング

最終段階でGAN（Generative Adversarial Network）損失を追加し、高周波ディテールを強化します。

```python
# Stage 3: GAN損失追加の概念コード
# Discriminatorを追加して敵対的学習

class VideoDiscriminator(torch.nn.Module):
    """動画品質を判定するDiscriminator（簡略版）"""
    def __init__(self):
        super().__init__()
        self.layers = torch.nn.Sequential(
            torch.nn.Conv3d(3, 64, kernel_size=4, stride=2, padding=1),
            torch.nn.LeakyReLU(0.2),
            torch.nn.Conv3d(64, 128, kernel_size=4, stride=2, padding=1),
            torch.nn.LeakyReLU(0.2),
            torch.nn.Conv3d(128, 1, kernel_size=4, stride=1, padding=1),
        )

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return self.layers(x)

# GAN損失をVAE損失に追加
# L_total = L_VAE + lambda_gan * L_GAN
# GAN損失により、ぼやけた再構成を防ぎ、
# 空間的な鮮明さと時間的な滑らかさのバランスを取る
```

**よくある間違い:**
> Stage 3のGAN損失を早い段階（Stage 1〜2）から導入すると、学習が不安定になりやすいことが報告されています。必ず段階的に損失を追加してください。VAE単体で十分な再構成品質を確保してからGAN損失を導入するのが定石です。

## 動画データセットを準備する

ファインチューニングの品質はデータセットの質に大きく依存します。ここでは動画データの収集から前処理、キャプション作成までの実践的な手順を紹介します。

### データ収集のベストプラクティス

コミュニティの複数のレポートに基づくと、以下のガイドラインが推奨されています。

| 項目 | 推奨値 | 備考 |
|------|--------|------|
| クリップ数 | 20〜30本 | 12-15本でも高品質なら可 |
| 解像度 | 256×256 | 学習時。推論は480p以上 |
| 長さ | 4〜8秒/クリップ | フレーム数は81程度 |
| フォーマット | MP4 (H.264) | 一般的なコーデック |
| 多様性 | ポーズ > 場所 | 同じ場所で異なるポーズが有効 |

**データ収集で注意すべき制約:**
- 強いフィルターやスタイライズは避ける（ベースモデルとの干渉を防ぐ）
- アップスケールした動画は使わない（ノイズパターンを学習してしまう）
- 露出のバランスを統一する（明暗の偏りを避ける）

### FFmpegによる動画前処理

収集した動画をモデルの学習に適した形式に変換します。

```bash
#!/bin/bash
# preprocess_videos.sh
# 動画をWan2.2ファインチューニング用に前処理するスクリプト

INPUT_DIR="./raw_videos"
OUTPUT_DIR="./processed_videos"
RESOLUTION="256"
FPS="24"
MAX_DURATION="8"

mkdir -p "$OUTPUT_DIR"

for video in "$INPUT_DIR"/*.mp4; do
    filename=$(basename "$video" .mp4)

    # リサイズ、FPS統一、長さ制限
    ffmpeg -i "$video" \
        -vf "scale=${RESOLUTION}:${RESOLUTION}:force_original_aspect_ratio=decrease,pad=${RESOLUTION}:${RESOLUTION}:(ow-iw)/2:(oh-ih)/2" \
        -r "$FPS" \
        -t "$MAX_DURATION" \
        -c:v libx264 \
        -preset medium \
        -crf 18 \
        -an \
        "$OUTPUT_DIR/${filename}.mp4"

    echo "Processed: ${filename}.mp4"
done

echo "Total processed: $(ls "$OUTPUT_DIR"/*.mp4 | wc -l) videos"
```

### キャプション作成の戦略

各動画クリップには対応する`.txt`キャプションファイルが必要です。キャプションの品質が学習結果を大きく左右します。

```text
# video_001.txt の例（良い例）
a woman walking through a park, natural lighting,
medium shot, slight camera movement, green trees in background

# video_001.txt の例（悪い例 - 詩的すぎる）
a beautiful ethereal goddess gracefully gliding through
an enchanted forest bathed in golden light
```

**キャプション作成のルール:**

1. **リテラル（直訳的）に書く**: 実際に映っているものをそのまま記述する
2. **構造化する**: 被写体 → アクション → カメラワーク → 背景の順
3. **スタイル語は控えめに**: 「beautiful」「amazing」などの主観的な形容詞は避ける
4. **トリガーワード戦略**: キャラクターLoRAの場合、`sora_person person`のように「インスタンストークン＋クラストークン」をキャプション先頭に配置する

### メタデータCSVの生成

DiffSynth-Studioでの学習にはメタデータCSVが必要です。

```python
# generate_metadata.py
# 動画ファイルとキャプションからメタデータCSVを生成

import csv
from pathlib import Path


def generate_metadata(
    video_dir: str,
    caption_dir: str,
    output_csv: str = "metadata.csv",
) -> int:
    """動画ディレクトリとキャプションディレクトリから
    学習用メタデータCSVを生成する

    Returns:
        生成されたエントリ数
    """
    video_path = Path(video_dir)
    caption_path = Path(caption_dir)
    entries = []

    for video_file in sorted(video_path.glob("*.mp4")):
        caption_file = caption_path / f"{video_file.stem}.txt"
        if not caption_file.exists():
            print(f"Warning: caption not found for {video_file.name}")
            continue

        prompt = caption_file.read_text(encoding="utf-8").strip()
        entries.append({"video": video_file.name, "prompt": prompt})

    with open(output_csv, "w", newline="", encoding="utf-8") as f:
        writer = csv.DictWriter(f, fieldnames=["video", "prompt"])
        writer.writeheader()
        writer.writerows(entries)

    print(f"Generated {len(entries)} entries in {output_csv}")
    return len(entries)


if __name__ == "__main__":
    generate_metadata("./processed_videos", "./captions")
```

| 正則化画像 | 1〜3倍 | 1〜3倍 | 学習画像数に対する比率 |

### VRAM使用量の最適化

VRAM不足は学習時の最大の課題です。以下の表はAMDの検証に基づく各構成のVRAM使用量です。

| モデル | 学習方式 | CPUオフロード | VRAM |
|--------|---------|-------------|------|
| 5B | LoRA | なし | 31.34 GB |
| 5B | LoRA | あり | 29.96 GB |
| 5B | Full | なし | 133.01 GB |
| 5B | Full | あり | 42.74 GB |
| 14B | LoRA | なし | 74.93 GB |
| 14B | Full | あり | 95.71 GB |

**制約条件:**
> LoRA学習でも5Bモデルは約30GBのVRAMが必要です。RTX 3090（24GB）では、`blocks_to_swap`オプションやCPUオフロードを活用しても、解像度やフレーム数を下げる必要がある場合があります。RTX 4090（24GB）やRTX 5090（32GB）が推奨環境です。

## 過学習を防ぐテクニックを実践する

動画生成モデルのファインチューニングでは過学習が大きな課題です。AMDのROCm Blogでの検証では、5エポック目で既に意図しないキャラクターが混入する現象が報告されています。

### 過学習の兆候と対策

| 兆候 | 原因 | 対策 |
|------|------|------|
| プロンプトと無関係なキャラクターが出現 | 学習データのスタイルを過度に記憶 | エポック数を減らす（3-5） |
| 背景が固定パターン化 | データの多様性不足 | 異なる背景の動画を追加 |
| 色味が不自然に偏る | データの露出バランスが偏り | 露出が均一なデータを選別 |
| 動きがぎこちなくなる | フレーム数不足 | 81フレーム以上のクリップを使用 |

### 正則化画像の活用

正則化（regularization）画像は、LoRAの過学習を防ぐための重要なテクニックです。

```bash
# 正則化画像の生成例
# ベースモデルでシンプルなプロンプトから生成

# アイデンティティLoRAの場合
# - 20〜60枚の正則化画像を用意
# - クラストークン（例: "person"）のみでキャプション
# - インスタンストークン（例: "sora_person"）は含めない
```

**ハマりポイント:**
> `guidance_scale`（CFG）はWan2.2では**3.5〜5.5**の範囲が推奨されています。Stable Diffusionで慣れた7.0〜12.0の値を使うと、過度にコントラストが強調された不自然な動画になりがちです。また、LoRA weightも1.0ではなく**0.5〜0.8**から始めることで、自然な結果が得られやすくなります。
